== Introduction — Les Trois MVP0 et l'Acteur Caché

=== Le symptôme

Depuis le 1er mai 2026, la définition du "MVP0" — le premier produit vendable —
a changé trois fois :

. *MVP0 = PlantUML SaaS* (v6) : un utilisateur paie, un diagramme est généré sur
  son repo GitHub. Priorité 0 = Stripe + OAuth. Le SaaS d'abord.
. *MVP0 = Capsule Vidéo* (v7) : le slider + TTS + FFmpeg → MP4. Priorité 0 =
  contenu pour soi avant le SaaS pour autrui. Le plugin d'abord.
. *MVP0 = Anonymiseur* (v15) : détection PII → datasets propres. Priorité 0 =
  nettoyage des données avant tout RAG. La donnée d'abord.

Puis une quatrième couche s'est ajoutée : l'Employé Synthétique par Silo de
Raffinement — fine-tuning incrémental sur bounded context DDD client.

Ces quatre définitions sont **toutes correctes**. Chacune capture une couche
réelle du produit. Mais aucune ne capture la couche **qui rend les trois autres
possibles** : le mécanisme par lequel le LLM accède au contexte augmenté sans
que l'outil de prompt engineering soit modifié.

=== Le diagnostic

Ce qui rend les quatre couches possibles, c'est une infrastructure commune :

. Un utilisateur (développeur, formateur, stagiaire) ouvre un outil de prompt
  engineering — opencode, Claude Code, Cursor, ou Copilot
. Le contexte de cet outil est **déjà augmenté** par le vecteur composite :
  RAG pgvector (similarité sémantique), Knowledge Graph Graphify (relations
  exactes), Modèle Métier SQL (DDD)
. Cette augmentation n'est pas faite par un plugin de l'outil — elle est faite
  par un **pipeline Gradle externe** qui injecte le contexte via stdout
. L'outil de prompt engineering n'est **jamais altéré**. Pas de fork, pas de
  plugin custom, pas d'extension propriétaire. Il consomme le contexte comme
  il consommerait n'importe quel stdout
. Le contrat entre le pipeline et l'outil est la **tâche Gradle typée** —
  paramètres validés à la compilation, output structuré, reproductible

C'est cette infrastructure qui est le **vrai produit**. Elle est le capital
fixe qui médiatise le vibecoding. Elle est antérieure à Edster, antérieure
à la capsule vidéo, antérieure à l'anonymiseur — elle les rend possibles
et les unifie.

=== La contradiction

Le workspace, par son propre développement, se heurte à ce que Gramsci appelle
l'équilibre instable des forces : les quatre couches produit (plugins,
capsule vidéo, SaaS, employé synthétique) sont des forces centrifuges qui
tirent le développement dans quatre directions. Mais le **capital fixe**
— le pipeline d'augmentation de contexte — est la force centripète qui
les maintient dans la même orbite.

La contradiction est la suivante : le pipeline est déjà opérationnel
(les plugins Gradle tournent, LangChain4j est intégré, le RAG est fonctionnel),
mais il n'est **pas encore packagé comme un produit conscient de lui-même**.
Il est le sol sur lequel les quatre autres couches marchent, mais il n'a pas
encore de nom, pas de spécification, pas de priorité propre.

Ce document résout cette contradiction en **donnant un nom et une priorité
absolue** au pipeline lui-même.

== Le Canal de Communication : stdout comme Protocole Universel

=== Le principe

Toute conversation entre un outil de prompt engineering et le pipeline
d'augmentation de contexte passe par un canal unique : **stdout**.

Ce choix n'est pas une limitation technique. C'est un choix architectural
délibéré qui garantit :

*Compatibilité universelle* ::
Tout outil de prompt engineering sait lire stdout. opencode lit stdout.
Claude Code lit stdout. Cursor lit stdout. Copilot lit stdout. Un terminal
humain lit stdout. Zéro intégration spécifique par outil.

*Zéro altération de l'outil* ::
Le pipeline ne modifie pas l'outil. Il ne lui injecte pas de plugin. Il ne
patche pas son runtime. Il ne hook pas ses APIs internes. Il émet sur stdout,
et l'outil consomme ce stdout comme il consommerait la sortie de n'importe
quelle commande shell.

*Déterministe et testable* ::
stdout est un flux de bytes. On peut le capturer, le parser, le valider,
le comparer entre deux runs. C'est le format le plus simple et le plus robuste
pour un contrat d'interface.

*Fonctionne en local et en CI/CD* ::
En local, stdout est immédiat. En CI/CD, stdout est le log du workflow.
Le même format, le même contrat, deux contextes d'exécution, zéro adaptation.

=== Les deux modes d'invocation

Le pipeline peut être invoqué selon deux modes, qui partagent le même canal
de sortie (stdout) mais diffèrent par leur mode de déclenchement :

==== Mode 1 — Invocation Directe par Tool LLM (Local)

[source]
----
UTILISATEUR (dans opencode)
    │  "Analyse le module auth et suggère des corrections"
    ▼
GEMMA 4 (orchestrateur opencode)
    │  Thinking: décomposer → choisir tâche Gradle appropriée
    ▼
OPEncode LANCE UNE TÂCHE GRADLE (via tool use)
    │  ./gradlew analyzeCode -Ptarget=auth-module
    ▼
PLUGIN GRADLE (Kotlin, LangChain4j)
    ├── Prépare le contexte : RAG pgvector → documents pertinents
    ├── Prépare le knowledge graph : Graphify → relations exactes
    ├── Assemble le vecteur composite : RAG + KG + SQL DDD
    ├── Appelle DeepSeek-v4-pro via LangChain4j avec le vecteur composite
    └── Parse la réponse → stdout structuré (JSON lines)
    ▼
OPEncode LIT LE STDOUT
    │  Le résultat apparaît dans la sortie standard de la session
    ▼
GEMMA 4 (orchestrateur opencode)
    │  Parse le stdout, visualise l'état, décide de la prochaine action
    ▼
UTILISATEUR
    "Voici l'analyse. Problème détecté : dépendance circulaire dans auth-module.
    Suggestion : extraire l'interface en module séparé. Lancer ./gradlew refacto ?"
----

Dans ce mode :
* L'utilisateur ne quitte jamais opencode
* Le LLM (Gemma 4) ne génère pas le code complexe — il route vers la tâche Gradle
* La tâche Gradle encapsule l'appel à DeepSeek-v4-pro avec le contexte augmenté
* Le résultat remonte en stdout, parsé par Gemma 4, affiché à l'utilisateur
* opencode n'est **pas modifié** — il utilise son mécanisme natif de tool use
  pour lancer `./gradlew`, exactement comme il lancerait `ls` ou `git status`

==== Mode 2 — Invocation par Stimulus CI/CD (Distant)

[source]
----
UTILISATEUR ou TRIGGER AUTOMATIQUE
    │  Commit poussé, PR ouverte, workflow_dispatch manuel
    ▼
GITHUB ACTIONS (ou GitLab CI, Gitea Actions)
    │  Le workflow est déclenché
    ▼
RUNNER EXÉCUTE LA TÂCHE GRADLE
    │  ./gradlew analyzeCode -Ptarget=auth-module
    ▼
PLUGIN GRADLE (Kotlin, LangChain4j)
    ├── Même pipeline que le mode 1 :
    ├── RAG pgvector + Graphify + SQL DDD → vecteur composite
    ├── Appel DeepSeek-v4-pro via LangChain4j
    └── stdout structuré (JSON lines)
    ▼
CI/CD CAPTURE LE STDOUT COMME LOG DE WORKFLOW
    │  Le stdout est l'artefact principal du job
    ▼
STIMULUS DE COMMUNICATION
    │  Le développeur, dans sa session opencode, déclenche :
    │  "Récupère l'output du dernier workflow sur auth-module"
    ▼
OPEncode LIT LE STDOUT DU WORKFLOW
    │  Via gh run view --log, ou via l'API GitHub, ou via un fichier artifacts
    │  Le résultat apparaît dans la sortie standard de la session opencode
    ▼
GEMMA 4 (orchestrateur opencode)
    │  Parse, visualise, décide de la suite
    ▼
UTILISATEUR
    "Workflow #42 terminé. Analyse complète dispo. Erreur identifiée, correctif
    suggéré dans le job log. Lancer le correctif ?"
----

Dans ce mode :
* Le déclenchement est un événement CI/CD (push, PR, schedule, dispatch)
* Le pipeline tourne sur un runner distant (GitHub Actions, infrastructure client)
* Le stdout est le log du job — l'artefact public du workflow
* Le développeur, dans sa session opencode, récupère ce stdout par un
  mécanisme de **stimulus** — une commande explicite ou un hook automatique
* La même augmentation de contexte, le même vecteur composite, mais le
  résultat est consommé **en différé** plutôt qu'en temps réel

=== Les trois modes sont le même produit, avec des réceptivités différentes

La différence entre les modes n'est pas une différence de produit.
C'est une différence de **réceptivité** — comment et quand le résultat
est consommé. Le contrat est identique : tâche Gradle typée → LangChain4j →
DeepSeek → stdout structuré. Mais chaque mode d'exécution a une capacité de
communication différente. Rien ne garantit l'interception synchrone.
Le plus bas dénominateur commun est le **mode asynchrone par commit + push**.

==== Mode 1 — Local Gradle (Synchrone)

[source]
----
opencode ──→ ./gradlew decomposeIntention -Pintention="..."
               │
               ▼  (même processus, stdout immédiat)
           [PLAN] EPIC-0: Skeleton (3 pts)
           [PLAN] EPIC-1: TTS integration (8 pts)
           ...
               │
               ▼
opencode lit le stdout en temps réel
→ l'humain valide ou corrige
----

*Réceptivité* : synchrone totale. Le stdout est lu immédiatement
par le processus appelant. Pas de latence, pas de perte.
Le LLM (Gemma 4) parse le stdout et visualise pour l'humain.

*Contrainte* : nécessite que l'humain soit devant opencode.
Ne scale pas en autonomie.

==== Mode 2 — CI/CD Runner (Asynchrone par Commit + Push, Semi-Interactif)

[source]
----
GITHUB ACTIONS (trigger: push, schedule, workflow_dispatch)
    │
    ▼
RUNNER exécute ./gradlew decomposeIntention -Pintention="..."
    │
    │  ⚠ PENDANT LE RUN, le runner PEUT être sensible aux push entrants
    │  (git fetch en boucle, polling léger)
    │
    ▼  (le stdout est le log du job — écoutable de l'extérieur)
LE PLUGIN GÉNÈRE LE PLAN → ÉCRIT LE FICHIER → COMMIT → PUSH
    │
    ▼
LE COMMIT EST LE SIGNAL
    │  "feat(plan): EPIC-0→4 pour capsule-video (21 pts)"
    │  Le git log devient l'historique des décisions
    │
    ▼
LE DÉVELOPPEUR (ou l'agent suivant)
    │  git pull → lit le commit → parse le plan → décide de la suite
    │
    ▼
opencode (réactif) : "Un nouveau plan a été commité. L'exécuter ?"
----

*Réceptivité* : asynchrone par commit + push, avec fenêtre de stimulation.
Le runner a deux visages :
. *Écoutable* — ses logs stdout/stderr sont lisibles en temps réel
  depuis l'extérieur (GitHub UI, `gh run watch`, API)
. *Semi-interactif* — dans son contexte d'exécution (le temps de vie du runner),
  le runner peut être sensible aux push entrants. Un `git fetch` en boucle
  permet à un autre agent de committer une instruction PENDANT que le runner
  tourne. Ce n'est pas une conversation continue — c'est une fenêtre de
  stimulation bornée par la durée du run.

*Limite fondamentale* : le runner n'a pas de boucle conversationnelle dans
son contexte d'exécution. Il peut juste recevoir des stimuli (commits) dans
sa fenêtre de vie et ajuster. Passé le timeout, le runner meurt. La seule
mémoire persistante est le commit.

==== Le Commit + Push Comme Protocole de Communication Asynchrone Universel

Le commit + push est le seul protocole qui fonctionne dans TOUS les modes :
local (git log), CI/CD (workflow → push), distant (autre machine),
cross-agent (agent A commit, agent B pull). Zéro middleware. Zéro queue.
Zéro webhook obligatoire. Juste Git.

*Pas de configuration* ::
Pas de webhook à enregistrer. Pas de secret à partager. Pas d'URL de callback.
Git push fonctionne partout, tout le temps, sans setup.

*Pas de perte de message* ::
Un webhook peut échouer si le receiver est down. Un commit est persistant —
il attend dans le repo que quelqu'un le pull. Pas de timeout, pas de retry.

*Audit intégré* ::
L'historique Git est l'historique des décisions. Qui a généré quoi, quand,
avec quel modèle, en combien de temps. Impossible à falsifier (SHA chain).

*Double lisibilité* ::
Le sujet du commit est pour l'humain. Le corps du commit est pour la machine
(JSON structuré). Les deux coexistent dans le même artefact.

*Boucle automatique* ::
Un push peut déclencher un nouveau workflow GitHub Actions (via `on: push`).
Le commit de sortie d'une tâche devient le déclencheur de la tâche suivante.

Convention de commit pour les tâches du pipeline :

[source]
----
<type>(<scope>): <description>

[PLAN]
EPIC-0: Skeleton Gradle (3 pts)
EPIC-1: TTS integration (8 pts)
...

[STATUS]
executed_by: ci-runner-42
model: deepseek-v4-pro
mode: MAX_THINK
duration_ms: 127043
timestamp: 2026-05-06T14:32:01Z
----

Le corps du commit contient le payload structuré. Le sujet est le résumé
lisible par l'humain. Le SHA est l'identifiant unique du message.

==== Les Trois Niveaux d'Asynchronie que Git Absorbe Naturellement

. *Asynchronie temporelle* — le runner exécute à 3h du matin (cron),
   le dev lit le résultat à 9h. Le commit attend.
. *Asynchronie spatiale* — le runner est sur GitHub Actions (US),
   le dev est en France. Le push traverse l'Atlantique.
. *Asynchronie outil* — le runner utilise Gradle, le dev utilise opencode,
   l'agent FPA utilise LangChain4j. Tous lisent Git. Aucun ne dépend
   du protocole des autres.

== Le Vecteur Composite de Contexte — La Matière Première de l'Augmentation

=== Structure

Le vecteur composite est l'assemblage de trois composantes de contexte que
la tâche Gradle injecte dans le prompt envoyé au LLM lourd (DeepSeek-v4-pro) :

[cols="1,2,2,3", options="header"]
|===
| Composante | Source | Outil | Apport
| **RAG pgvector** | `office/` (data pédagogique) + `OSS/`
(code source public) | Embeddings ONNX (AllMiniLmL6V2) + PostgreSQL pgvector |
*Similarité sémantique* — le "flou productif" qui trouve les documents
pertinents même sans mot-clé exact
| **Knowledge Graph** | `graph.json` (produit par graphify-gradle
sur tout l'écosystème) | Graphify : nœuds (fichiers, projets), arêtes
(imports, dépendances, références), communautés | *Relations exactes* —
le déterministe structurant qui montre comment les composants se connectent
| **Modèle Métier SQL** | DDL (schéma entité-relation) → bounded
contexts DDD | Parser SQL → extraction des agrégats, value objects, entités |
*Domaine métier* — la "troisième voie de contexte" qui donne au LLM la
structure du métier du client
|===

=== Assemblage dans la tâche Gradle

[source,kotlin]
----
// Pseudo-code de la tâche Gradle qui assemble le vecteur composite
tasks.register<AnalyzeCodeTask>("analyzeCode") {
    target.set(project.findProperty("target") as String)

    doLast {
        // 1. RAG — chercher les documents pertinents par similarité
        val ragContext = pgvector.query(
            embedding = onnx.embed(query),
            topK = 10,
            filter = ZoneFilter(Zone.OSS + Zone.OFFICE_VISION)
        )

        // 2. Knowledge Graph — naviguer les relations exactes
        val kgContext = graphify.query(
            predicate = "node.path CONTAINS '$target'",
            depth = 2,  // voisins jusqu'à distance 2
            edgeTypes = listOf("imports", "depends_on", "references")
        )

        // 3. Modèle Métier SQL — extraire les bounded contexts
        val dddContext = sqlParser.extractBoundedContexts(
            ddl = schemaContent,
            target = target
        )

        // 4. Assemblage du vecteur composite
        val compositeContext = CompositeContext(
            rag = ragContext,       // similarité sémantique
            knowledgeGraph = kgContext,  // relations exactes
            domainModel = dddContext     // modèle métier DDD
        )

        // 5. Injection dans le prompt LangChain4j
        val prompt = PromptTemplate.from("""
            Contexte augmenté :
            {{compositeContext}}

            Tâche : {{userRequest}}
        """).apply(compositeContext).apply(userRequest)

        // 6. Appel au LLM lourd avec contexte augmenté
        val response = deepseek.generate(prompt)

        // 7. Sortie en stdout structuré
        println("[CTX] composite_vector_size=${compositeContext.size}")
        println("[DS_STATUS] ${response.metadata}")
        println(response.content)
    }
}
----

=== Ce que le vecteur composite permet

*Pas de recherche manuelle* ::
Le développeur ne fait pas de `grep`, de `find`, de navigation dans l'arborescence.
La tâche Gradle a déjà trouvé les 10 documents les plus pertinents, les 20 relations
de dépendance critiques, et le bounded context DDD du module cible.

*Contexte ancré et vérifiable* ::
Le RAG est ancré sur les documents réels du workspace — pas sur une hallucination.
Le knowledge graph est produit par un parseur déterministe — pas par un LLM.
Le modèle métier SQL est le schéma réel de la base — pas une spéculation.

*Économie de tokens massive* ::
Le vecteur composite remplace 500+ lignes de contexte EAGER/LAZY par ~50 tokens
ciblés. Ratio d'économie documenté : 71x (v10, EPIC 9).

*Transférable entre projets* ::
Le même vecteur composite, assemblé par les mêmes tâches Gradle, peut être injecté
dans opencode, dans un plugin slider-gradle, dans un pipeline CI/CD, ou dans le
futur SaaS Edster. Le contrat est le stdout. Le format est JSON lines. Le pont
est LangChain4j.

== Les Deux Acteurs du Pipeline

=== Acteur 1 — Le Cerveau Léger (Orchestrateur Local)

L'orchestrateur est un petit modèle rapide (Gemma 4 E4B, 4.5B effectif) qui tourne
dans l'outil de prompt engineering (opencode).

Son rôle :
* Router les requêtes utilisateur vers la tâche Gradle appropriée
* Lancer `./gradlew <tâche>` via le mécanisme natif de tool use de l'outil
* Parser le stdout de la tâche
* Visualiser l'état et le résultat à l'utilisateur
* Décider de la prochaine action (thinking mode)

Ce qu'il ne fait **pas** :
* Générer du code complexe
* Analyser en profondeur
* Synthétiser des informations cross-projets
* Maintenir le contexte EAGER/LAZY au-delà du strict nécessaire

=== Acteur 2 — Le Cerveau Lourd (Exécuteur via LangChain4j)

L'exécuteur est un gros modèle frontalier (DeepSeek-v4-pro, 49B activés, 1M contexte)
qui tourne sur Ollama Cloud et est appelé **depuis le plugin Gradle** via LangChain4j.

Son rôle :
* Générer du code complexe (refacto, nouveaux endpoints, tests)
* Analyser l'architecture en profondeur (dépendances, patterns, anti-patterns)
* Synthétiser des informations multi-fichiers, multi-projets
* Corriger des bugs non triviaux avec analyse de stacktrace

Ce qu'il ne fait **pas** :
* Interagir directement avec l'utilisateur
* Décider quelle tâche lancer — c'est le rôle de Gemma 4
* Servir du HTML ou du markup UI — c'est le rôle des templates Kotlin

== Pourquoi CECI est le Vrai Produit

=== Antériorité logique

Le pipeline d'augmentation de contexte est **logiquement antérieur** à toutes les
autres couches produit :

* Sans pipeline, les plugins Gradle sont des transformateurs sans contexte —
  ils transforment mais ne comprennent pas *quoi*
* Sans pipeline, la capsule vidéo génère des slides sans savoir quel matériel
  pédagogique existe dans `office/`
* Sans pipeline, le SaaS Edster provisionne des workspaces vides — des coquilles
  sans contenu augmenté
* Sans pipeline, l'employé synthétique ne peut pas être fine-tuné sur le bounded
  context DDD du client — il n'a pas de mécanisme d'observation du domaine

Le pipeline est la **condition de possibilité** des quatre couches. Il ne les
remplace pas — il les rend possibles. Et parce qu'il les rend possibles, il est
le produit le plus fondamental.

=== Universalité de l'outillage

Le pipeline ne dépend d'aucun outil de prompt engineering spécifique :

| Outil | Comment il consomme le pipeline |
| opencode | `./gradlew <tâche>` via tool use natif → stdout |
| Claude Code | `./gradlew <tâche>` via bash tool → stdout |
| Cursor | Terminal intégré → `./gradlew <tâche>` → stdout |
| Copilot | Terminal ou CI/CD → `./gradlew <tâche>` → stdout |
| Terminal humain | `./gradlew <tâche>` → stdout lu par l'humain |

Le contrat est le même partout : une commande shell, un stdout structuré, zéro
intégration spécifique. Le pipeline ne fait pas de favoritisme. Il émet sur stdout,
et tout outil capable de lancer une commande shell peut le consommer.

Cette universalité est un **argument commercial massif** :
* Pas de vendor lock-in sur l'outil de prompt engineering
* Pas de migration si l'utilisateur change d'outil
* Pas de courbe d'apprentissage — `./gradlew <tâche>` est la seule commande
* Zéro configuration par outil — le pipeline est un exécutable, pas un plugin

=== Le capital fixe comme produit

Dans la dialectique matérialiste, le capital fixe est le moyen de production qui
ne se consomme pas dans l'acte productif — il le médiatise et le rend possible.
La machine-outil qui usine les pièces. Le pipeline CI/CD qui build le code.

Le pipeline d'augmentation de contexte est le **capital fixe** du vibecoding :

* Il ne génère pas le code — il **prépare les conditions** pour que le LLM
  puisse le générer correctement
* Il ne publie pas le contenu — il **structure le contexte** pour que le
  contenu généré soit pertinent
* Il ne facture pas le client — il **rend le service tellement utile** que
  le client veut payer pour y accéder

La valeur du capital fixe ne se mesure pas à sa consommation (combien de tokens
le pipeline utilise), mais à sa **productivité marginale** (combien de valeur
chaque appel au LLM produit grâce au contexte augmenté). C'est la différence entre :

* "Écris un microservice de gestion de commandes" → 50% de code pertinent
* "Écris un microservice de gestion de commandes [avec contexte : voici les
  10 fichiers les plus pertinents + le graphe de dépendances + le modèle métier
  SQL du domaine commandes]" → 95% de code pertinent

Le pipeline transforme le rendement du LLM sans consommer plus de compute LLM.
C'est la définition même de la productivité du capital.

=== Convergence de canaux

Le pipeline est le **point de convergence** de tous les canaux de développement
identifiés dans l'architecture :

[source]
----
CANAL EAGER/LAZY (contexte statique, chargé en début de session)
    └→ Limité à ~800 tokens, figé, pas de mise à jour en cours de session

CANAL RAG (contexte dynamique, chargé à la demande)
    └→ Requêtes vectorielles sur pgvector, embedding ONNX

CANAL GRAPHIFY (contexte structurel, relations exactes)
    └→ Knowledge graph transverse, requêtable, navigable

CANAL CI/CD (contexte distant, asynchrone)
    └→ Logs de workflow, artifacts, statuts de build
                           │
    Tous les canaux ──────►│
                           ▼
            VECTEUR COMPOSITE (assemblé par tâche Gradle)
                           │
                           ▼
            LANGCHAIN4J → pro prompt → DeepSeek-v4-pro
                           │
                           ▼
            STDOUT (consommable par tout outil)
----

Avant le pipeline, chaque canal est un silo. Le EAGER/LAZY ne sait pas ce que le
RAG a trouvé. Le RAG ne sait pas ce que Graphify a structuré. Le CI/CD ne sait pas
que l'utilisateur est en train de travailler sur le même module en local.

Avec le pipeline, les quatre canaux convergent dans le vecteur composite, qui est
**injecté dans un seul prompt** au LLM lourd. Le LLM ne voit pas quatre sources
de contexte partielles — il voit **un système unifié** dont la structure lui
indique ce qu'il doit faire ensuite.

=== Le raisonnement dialectique

Le workspace suit un développement par contradictions successives :

. *Thèse* (1er mai) : le produit est Edster SaaS — une plateforme de provisionnement
. *Antithèse* (2 mai) : non, le produit est les plugins Gradle — ils font déjà tout

. *Synthèse partielle* (3-4 mai) : le produit est une méthode reproductible
  (anonymiseur → datasets → RAG → fine-tuning). La data d'abord

Mais cette synthèse cache une contradiction plus profonde : les plugins, le SaaS,
l'anonymiseur, la capsule vidéo — **tous** présupposent un mécanisme d'injection
de contexte augmenté. Ce mécanisme est déjà partiellement construit (RAG, Graphify,
LangChain4j), mais il n'est pas conscient de lui-même comme produit.

La v23 résout cette contradiction en **nommant** ce qui était implicite :

> Le produit n'est pas ce que le pipeline génère. Le produit est le pipeline
> lui-même — le capital fixe qui transforme n'importe quel outil de prompt
> engineering en coproducteur IA conscient du codebase, de la documentation,
> et du modèle métier du projet.

== Contrat d'Interface Formel du Pipeline

=== La tâche Gradle comme contrat

Chaque tâche du pipeline respecte un contrat formel :

[example]
.Contrat d'une tâche Gradle d'augmentation de contexte
====
*Entrée* — paramètres typés (`-P<key>=<value>`, validés à la compilation)
*Pré-processing* — Assemblage du vecteur composite (RAG + KG + SQL → contexte unifié)
*Exécution* — Appel LangChain4j → DeepSeek-v4-pro avec le vecteur composite injecté
*Post-processing* — Parsing de la réponse (Jackson → data class Kotlin)
*Sortie* — stdout structuré en JSON lines avec 4 types de lignes :
  `[CTX]` — métadonnées du contexte augmenté (taille, sources, filtres)
  `[DS_STATUS]` — statut de l'appel DeepSeek (RUNNING, DONE, FAILED, RETRY)
  `[RESULT]` — contenu de la réponse (JSON payload)
  `[ERROR]` — diagnostic d'erreur (stacktrace, suggestion)
*Code de retour* — 0 = succès, 1 = échec LLM, 2 = échec validation, 3 = timeout
====

=== Le format stdout unifié

Chaque plugin qui participe au pipeline émet son stdout selon ce format :

[source,json]
----
[CTX] {"vector_size": 15420, "sources": ["office/", "OSS/"], "components": ["rag", "kg", "sql"], "tokens_saved": 471}
[DS_STATUS] {"model": "deepseek-v4-pro", "mode": "MAX_THINK", "timestamp": "2026-05-06T09:32:15Z", "status": "RUNNING"}
[DS_STATUS] {"model": "deepseek-v4-pro", "mode": "MAX_THINK", "timestamp": "2026-05-06T09:34:22Z", "status": "DONE", "duration_ms": 127043}
[RESULT] {"summary": "Analyse du module auth-module terminée. 3 problèmes détectés.", "issues": [...], "suggestions": [...]}
[ERROR] null
----

Ce format est :
* *Parsable* par n'importe quel parseur JSON lines (`jq`, `grep`, script Python/Kotlin)
* *Filterable* par Gemma 4 — il peut extraire `[RESULT]` lignes pour l'utilisateur
* *Auditable* — le `[CTX]` ligne montre exactement quel contexte a été injecté
* *Versionnable* — le format peut évoluer (nouveaux types de lignes) sans casser
  les consommateurs existants

=== Le contrat d'invocation

L'utilisateur (via opencode ou n'importe quel outil) invoque le pipeline avec :

[source,bash]
----
./gradlew <tâche> -P<paramètres typés>
----

Exemples :

[source,bash]
----
# Analyse d'un module avec contexte augmenté complet
./gradlew analyzeCode -Ptarget=auth-module -Pmode=composite

# Génération de tests avec contexte RAG uniquement
./gradlew generateTests -Ptarget=payment-service -Pcontext=rag

# Refacto avec vecteur composite complet
./gradlew refactoModule -Ptarget=user-module -Pcontext=composite -Pstrategy=safe

# Rapport multi-module avec knowledge graph uniquement
./gradlew buildReport -Pscope=workspace -Pcontext=kg
----

Le paramètre `-Pcontext` contrôle le niveau d'augmentation :
* `composite` = RAG + KG + SQL (vecteur complet)
* `rag` = RAG uniquement (similarité sémantique)
* `kg` = Knowledge Graph uniquement (relations exactes)
* `sql` = Modèle métier SQL uniquement (DDD)
* `none` = Aucun contexte augmenté (prompt nu)

== Le Pipeline Comme Mécanisme de Perception du Système

=== Au-delà de la génération

Le pipeline d'augmentation de contexte ne se contente pas d'améliorer la qualité
des réponses du LLM. Il transforme le LLM de **générateur textuel** en **mécanisme
de perception du système** :

* *Perception spatiale* : le LLM sait dans quel cercle de confiance se trouve
  chaque fichier (workspace/ vs office/ vs OSS/ vs CSS/)
* *Perception relationnelle* : le LLM connaît le graphe de dépendances entre
  composants — qui importe quoi, qui référence qui
* *Perception métier* : le LLM comprend le modèle de domaine (bounded contexts,
  agrégats, value objects) via le schéma SQL parsé en DDD
* *Perception historique* : le LLM peut interroger l'archive de vision pour
  comprendre comment la stratégie a évolué

Cette perception est la condition du **vibecoding dialectique** théorisé dans
`BOOM_BAP.adoc` : l'intention (le vibe) n'est pas injectée dans le vide —
elle est injectée dans un système dont le LLM perçoit la structure. Le build vert
n'est pas un hasard — c'est le produit de cette perception augmentée.

=== La boucle d'apprentissage

[source]
----
SESSION OPEncode (Gemma 4)
    │  L'utilisateur exprime un besoin
    ▼
TÂCHE GRADLE (assembleCompositeContext)
    │  RAG + KG + SQL → vecteur composite
    ▼
DEEPSEEK-V4-PRO (via LangChain4j)
    │  Génère avec contexte augmenté
    ▼
STDOUT STRUCTURÉ
    │  [CTX] + [DS_STATUS] + [RESULT]
    ▼
GEMMA 4 LIT LE STDOUT
    │  Visualise, décide de la suite
    ▼
UTILISATEUR VALIDE OU CORRIGE
    │  Feedback humain dans la boucle
    ▼
SESSION SUIVANTE
    │  Le contexte augmenté inclut le résultat de la session précédente
    │  La mémoire (pgvector) est enrichie du nouvel état
    ▼
LE SYSTÈME APPREND
    └→ Boucle recommence, contexte plus riche, génération plus pertinente
----

Chaque itération enrichit la base de connaissance (pgvector), met à jour le
knowledge graph (Graphify delta), et affine le modèle métier (SQL import).
Le pipeline ne produit pas seulement des artefacts — il **produit de la
connaissance sur lui-même**.

C'est ce que `BOOM_BAP.adoc` appelle la contradiction motrice : la connaissance
est toujours en retard sur le réel, mais c'est cette course qui génère la valeur.
Le pipeline est le mécanisme qui rend cette course possible — et observable par
le LLM lui-même.

== Le Pipeline et la Gouvernance Agent

=== La gouvernance agent comme premier consommateur

Le pipeline d'augmentation de contexte est le **consommateur naturel** de la
gouvernance agent (`.agents/`, `PROMPT_REPRISE`, `INDEX.adoc`, `SESSION_CHECKLIST`).

Aujourd'hui, la gouvernance agent est chargée manuellement : Gemma 4 (ou
DeepSeek directement) lit `PROMPT_REPRISE.adoc`, `INDEX.adoc`, et les documents
EAGER au début de chaque session.

Avec le pipeline, la gouvernance agent est **injectée automatiquement** dans
le vecteur composite :

* `PROMPT_REPRISE.adoc` → composante "contexte de mission" du vecteur
* `INDEX.adoc` → composante "état du projet" (requêtable via RAG)
* `AGENT_GOVERNANCE.adoc` → règles EAGER injectées comme contraintes
* `SESSIONS_HISTORY.adoc` → historique consultable via RAG

Le pipeline ne remplace pas la gouvernance agent — il l'**automatise**.
Ce qui était chargé manuellement (et donc oubliable) devient injecté
automatiquement (et donc inoubliable).

=== Le pipeline rend la gouvernance agent exportable

Aujourd'hui, adopter la gouvernance agent signifie :
. Cloner un dépôt avec `.agents/`
. Comprendre la structure EAGER/LAZY
. Configurer manuellement le chargement de contexte
. Maintenir manuellement les INDEX.adoc

Avec le pipeline, adopter la gouvernance agent signifie :
. Cloner un dépôt avec `.agents/`
. Lancer `./gradlew assembleCompositeContext`
. Le contexte est injecté automatiquement → l'outil de prompt engineering
  est immédiatement augmenté

La barrière à l'adoption passe de "lire 500 lignes de doc et configurer
manuellement" à "une commande Gradle". C'est le même saut que Magic Stick
pour l'environnement de travail : de "configurer son poste pendant 2 jours"
à "boot sur la clé, tout est prêt".

== La Bidirectionnalité du Pipeline — opencode Comme Data Source Autant Que Consommateur

Le flux n'est pas unidirectionnel (`pipeline → opencode`). Le pipeline peut
aussi **extraire** la gouvernance agent d'opencode comme data source pour
LangChain4j. C'est la boucle complète : opencode consomme le contexte augmenté,
ET opencode **alimente** le pipeline en données de gouvernance structurée.

=== Les trois capacités d'extraction

==== Capacité 1 — Relocalisation du Texte des Agents dans LangChain4j

Chaque fichier `.agents/` contient du texte conçu pour être lu par un LLM
dans opencode. Mais ce même texte peut être **relocalisé** comme composante
du prompt LangChain4j dans une tâche Gradle :

[source,kotlin]
----
tasks.register<RelocalizeAgentsTask>("relocalizeAgents") {
    projectDir.set(file("."))
    agentDocs.set(listOf("AGENT.adoc", "PROMPT_REPRISE.adoc", "INDEX.adoc"))

    doLast {
        // 1. Parser les fichiers .agents/ en structured data
        val agentContext = AgentDocParser.parse(projectDir.get().asFile) {
            section("Règle 1") { type = AgentDocType.CONSTRAINT }
            section("Règle 2") { type = AgentDocType.PROCEDURE }
            section("Règle 2bis") { type = AgentDocType.EPISTEMIC_FILTER }
            section("Mission") { type = AgentDocType.MISSION }
            section("Portefeuille Transverse") { type = AgentDocType.CROSS_REF }
        }

        // 2. Injecter comme composante du vecteur composite
        val composite = CompositeContext(
            rag = pgvector.query(embedding),
            knowledgeGraph = graphify.query(predicate),
            domainModel = sqlParser.extractBoundedContexts(ddl),
            agentGovernance = agentContext  // ← NOUVEAU : gouvernance agent
        )

        // 3. Appel LangChain4j avec gouvernance injectée
        val prompt = PromptTemplate.from("""
            [GOUVERNANCE AGENT]
            Règles absolues : {{composite.agentGovernance.constraints}}
            Procédures : {{composite.agentGovernance.procedures}}
            Filtre épistémique : {{composite.agentGovernance.epistemicFilter}}
            Mission : {{composite.agentGovernance.mission}}
            Références croisées : {{composite.agentGovernance.crossReferences}}

            [CONTEXTE AUGMENTÉ]
            {{composite.rag}}
            {{composite.knowledgeGraph}}
            {{composite.domainModel}}

            [TÂCHE]
            {{userRequest}}
        """).apply(composite).apply(userRequest)

        println("[CTX] agent_docs_parsed=${agentContext.size}")
        val response = deepseek.generate(prompt)
        println(response.content)
    }
}
----

Ce qui change : au lieu que Gemma 4/DeepSeek lise les fichiers `.agents/`
dans le contexte opencode (~800 tokens EAGER + LAZY à la demande), c'est
une **tâche Gradle** qui les parse, les structure, et les injecte dans
le prompt LangChain4j de manière typée. Le LLM lourd (DeepSeek) n'a pas
besoin de "lire" la gouvernance agent — elle fait partie de son vecteur
composite au même titre que le RAG, le knowledge graph, et le modèle métier.

==== Capacité 2 — Initialisation de la Gouvernance Agent sur un Projet Vierge

Un nouveau projet opencode n'a pas de `.agents/`. Le pipeline peut
**générer** la structure de gouvernance initiale depuis les templates
référentiels du workspace :

[source,kotlin]
----
tasks.register<InitAgentGovernance>("initAgentGovernance") {
    projectDir.set(file("."))
    projectName.set(project.findProperty("name") as String)
    templateSource.set(layout.projectDirectory.dir("../codebase-gradle/templates/agents"))

    doLast {
        // 1. Copier les templates depuis le référentiel
        val templates = mapOf(
            "AGENT.adoc" to Template("AGENT_TEMPLATE.adoc"),
            ".agents/INDEX.adoc" to Template("INDEX_TEMPLATE.adoc"),
            ".agents/PROMPT_REPRISE.adoc" to Template("PROMPT_REPRISE_TEMPLATE.adoc"),
            ".agents/SESSION_CHECKLIST.adoc" to Template("SESSION_CHECKLIST_TEMPLATE.adoc")
        )

        templates.forEach { (path, template) ->
            val rendered = template.render(mapOf(
                "projectName" to projectName.get(),
                "date" to LocalDate.now().toString(),
                "license" to "Apache 2.0"
            ))
            File(projectDir.get().asFile, path).writeText(rendered)
        }

        // 2. Demander à LangChain4j d'affiner le template selon le contexte projet
        val projectContext = """
            Projet : ${projectName.get()}
            Stack : ${detectStack(projectDir.get().asFile)}
            Fichiers existants : ${listFiles(projectDir.get().asFile)}
        """

        val refined = langchain4j.generate(PromptTemplate.from("""
            Voici un projet sans gouvernance agent. Le template initial a été généré.
            Analyse le codebase existant et suggère des ajustements pour :
            - Le contexte EAGER (quels fichiers sont critiques au démarrage ?)
            - Les codebases de référence LAZY (ce projet dépend-il de plugins Gradle existants ?)
            - Les règles de vérification SESSION_CHECKLIST spécifiques à ce projet

            Template généré : $templates
            Contexte projet : $projectContext
        """))

        // 3. Appliquer les refinements suggérés par le LLM
        applyRefinements(projectDir.get().asFile, refined)

        println("[CTX] governance_initialized=true project=${projectName.get()}")
        println("[RESULT] ${refined.summary}")
    }
}

=== Les Tâches d'Audit et de Refactorisation avec Contraintes Fortement Typées

Le quatrième usage de la bidirectionnalité est la capacité à lancer des
tâches d'audit/analyse sur la gouvernance agent elle-même — puis à
refactoriser le comportement de l'agent (les fichiers `.agents/`) en
passant par LangChain4j, mais cette fois avec des **contraintes
structurantes fortement typées** par Gradle.

==== Le problème résolu

Aujourd'hui, quand opencode (Gemma 4) modifie un fichier `.agents/`, il
le fait avec un prompt conversationnel. Aucune validation structurelle.
Rien ne garantit que la modification respecte :
* La classification Vision/Opinion/Stratégie (Règle 2bis)
* Le format des sections EAGER/LAZY
* La cohérence du portefeuille transverse (tous les INDEX.adoc se connaissent)
* L'absence de PII dans une section publiable

Avec les tâches Gradle d'audit → refacto, le pipeline **audite** d'abord
(analyse), puis **refactorise** (génération LangChain4j), puis **valide**
(contraintes Kotlin typées), puis **réécrit** les fichiers `.agents/`.

==== La chaîne complète

[source]
----
ÉTAPE 1 — AUDIT (analyse déterministe + LLM)
    │  ./gradlew auditAgentGovernance
    │  Parse .agents/ → détecte :
    │    - Sections VISION sans tag [VISION]
    │    - Sections OPINION dans PROMPT_REPRISE (violation Règle 2bis)
    │    - INDEX.adoc qui référence un projet non listé dans le portefeuille transverse
    │    - AGENT.adoc qui ne mentionne pas la Règle 2bis
    │    - PII résiduelle dans une section taguée Niveau 3
    ▼
ÉTAPE 2 — RAPPORT D'AUDIT (stdout structuré)
    │  [AUDIT] issues=7 critical=2 warnings=5
    │  [AUDIT] issue_1: "PROMPT_REPRISE.adoc:45 — opinion non taguée"
    │  [AUDIT] issue_2: "AGENT.adoc — manque référence Règle 2bis"
    ▼
ÉTAPE 3 — REFACTO VIA LANGCHAIN4J (proposition)
    │  ./gradlew refactoAgentGovernance --auto-fix
    │  Pour chaque issue d'audit, appel LangChain4j → DeepSeek-v4-pro
    │  avec le contexte complet (RAG + KG + gouvernance agent actuelle)
    │  ET avec des contraintes structurantes Kotlin :
    │
    │  data class RefactoConstraint(
    │      val preserveHistory: Boolean = true,   // jamais supprimer [v1]...[vN]
    │      val tagOpinion: Boolean = true,         // ajouter // @OPINION
    │      val neverPublishable: List<String>,     // sections interdites en blog
    │      val mustBeVersionned: List<String>,     // sections à versionner
    │      val zoneFilter: Zone = Zone.OFFICE      // quelle zone cible
    │  )
    ▼
ÉTAPE 4 — VALIDATION KOTLIN TYPÉE (avant écriture)
    │  Avant d'écrire le fichier modifié, le plugin Gradle valide :
    │    - La structure AsciiDoc est intacte
    │    - Les tags [vN] sont préservés
    │    - Les sections @OPINION ne contiennent pas d'URL publique
    │    - Les références croisées sont valides (INDEX.adoc cohérent)
    │    - Le format stdout est respecté
    │  Si validation FAIL → rollback, rapport d'erreur
    │  Si validation PASS → écriture
    ▼
ÉTAPE 5 — ÉCRITURE ET COMMIT
    │  Les fichiers .agents/ sont modifiés
    │  git diff montre exactement ce qui a changé
    │  L'humain valide visuellement avant commit
    ▼
ÉTAPE 6 — RÉAUDIT (vérification post-refacto)
    │  ./gradlew auditAgentGovernance
    │  Vérifie que les issues précédentes sont résolues
    │  Aucune régression
----

==== Contrat d'une tâche d'audit → refacto

[example]
.Contrat de la tâche `refactoAgentGovernance`
====
*Entrée* — Fichiers `.agents/` du projet cible + rapport d'audit précédent
*Contraintes Kotlin typées* — `RefactoConstraint` (data class immuable, validée
  à la compilation) : preservationHistory, tagOpinion, neverPublishable, zoneFilter
*Appel LangChain4j* — DeepSeek-v4-pro avec :
  - Le vecteur composite complet (RAG + KG + SQL + gouvernance actuelle)
  - Les issues d'audit
  - Les contraintes structurantes Kotlin (injectées en system prompt)
*Validation pré-écriture* — `AgentDocValidator` : parse le document proposé,
  vérifie les contraintes, valide la cohérence cross-références
*Sortie* — Fichiers `.agents/` modifiés + stdout d'audit post-refacto

====

Imaginons un `PROMPT_REPRISE.adoc` qui contient :

Le pipeline n'est pas un injecteur de contexte unidirectionnel. C'est une
**interface bidirectionnelle** entre la gouvernance agent (opencode) et
l'exécution LLM (LangChain4j dans tâche Gradle) :

Les six tâches Gradle forment un système complet :

[cols="1,3,2", options="header"]
|===
| Tâche | Direction | Ce qu'elle fait
| `assembleCompositeContext` | pipeline → opencode | Injecte le vecteur composite
dans le stdout → opencode lit le contexte augmenté
| `relocalizeAgents` | opencode → pipeline | Parse `.agents/` → injecte la
gouvernance comme composante du vecteur composite dans LangChain4j
| `initAgentGovernance` | pipeline → opencode | Crée `.agents/` sur un projet
vierge à partir des templates + raffinement LLM
| `importOpenCodeGovernance` | opencode → pipeline | Absorbe la gouvernance
agent d'un projet externe → vectorisation + génération build.gradle.kts
| `auditAgentGovernance` | opencode → pipeline | Analyse `.agents/` →
détection issues (PII, opinions non taguées, incohérences)
| `refactoAgentGovernance` | pipeline → opencode | Corrige les issues d'audit
via LangChain4j avec contraintes Kotlin typées → écriture validée
|===

L'ensemble constitue le **métabolisme complet de la gouvernance agent** :
naissance (init), absorption (import), structuration (relocalize), diagnostic
(audit), correction (refacto), et augmentation (assemble). Le LLM n'est plus
un lecteur passif de fichiers `.adoc` — il est le **gestionnaire actif** de
son propre cycle de vie de gouvernance.

== codex — Le Plugin d'Acquisition de Documents du Pipeline

=== Positionnement dans l'architecture

Le vecteur composite repose sur trois composantes : RAG pgvector, Knowledge Graph
Graphify, Modèle Métier SQL. Ces trois composantes sont alimentées par des documents
structurés — mais aujourd'hui, l'alimentation est manuelle ou supposée faite en
amont. `codex` est le **plugin Gradle qui industrialise l'acquisition** de ces
documents, en particulier depuis les sources non-structurées comme les PDFs.

Sans `codex`, le pipeline d'augmentation de contexte ne peut pas ingérer
`office/books-collection/` ni aucune source documentaire dense. Le RAG est
borgne sur la connaissance livresque. `codex` est le **sens de la lecture**
du pipeline.

=== Organisation physique — monorepo, indépendance logique de build

==== Le principe

`codex` adopte un pattern **monorepo physique, indépendance logique de build** :
un seul dépôt git, plusieurs projets Gradle autonomes cohabitant dans le même
dossier racine — sans jamais être déclarés dans le `settings.gradle.kts` de `codex`.

Ce pattern est une troisième voie entre :

* *Multi-module Gradle* (A) — un `settings.gradle.kts` racine qui déclare tous
  les sous-projets via `include(":codex-pdf")`. Couplage fort de build, impossible
  de builder un sous-module indépendamment.
* *Dépôts totalement séparés* (B) — chaque plugin dans son propre dépôt git.
  Cohérent avec le pattern workspace existant (`plantuml-gradle`, `graphify-gradle`),
  mais fragmente l'historique et alourdit la gouvernance pour un écosystème
  fortement couplé sémantiquement.

La troisième voie : **un seul `.git` à la racine de `codex/`, des
`settings.gradle.kts` indépendants par sous-projet, `codex` dictant la version**.

==== `codex` est le contrat

`codex` n'est pas un plugin parent abstrait séparé de son contrat.
**`codex` est le contrat** — il contient directement les types partagés,
l'interface `DocExtractor`, et les conventions communes (`StdoutFormatter`,
groupe Maven). Il n'existe pas de module `doc-extractor` séparé.

`codex-pdf` et `codex-epub` importent `codex` comme unique dépendance de contrat.

==== Structure du dépôt

[source]
----
OSS/
  codex/                        ← .git ici — un seul repo git
    build.gradle.kts            ← codex = le contrat + conventions communes
    settings.gradle.kts         ← rootProject.name = "codex" UNIQUEMENT, zéro include()
    gradle.properties           ← VERSION UNIQUE : codex.version=1.0.0
    src/
      main/kotlin/
        DocExtractor.kt         ← interface + data classes (BlockType, ExtractedBlock...)
        StdoutFormatter.kt      ← format [CTX][DS_STATUS][RESULT][ERROR] partagé
    codex-pdf/
      build.gradle.kts          ← projet Gradle autonome
      settings.gradle.kts       ← rootProject.name = "codex-pdf" UNIQUEMENT
      src/
    codex-epub/
      build.gradle.kts          ← projet Gradle autonome
      settings.gradle.kts       ← rootProject.name = "codex-epub" UNIQUEMENT
      src/
----

==== `codex` dicte la version

La version est déclarée **une seule fois** dans `codex/gradle.properties` :

[source,properties]
----
codex.version=1.0.0
----

Chaque sous-projet lit cette version via un chemin relatif dans son
`build.gradle.kts` — pas de duplication, pas de drift possible :

[source,kotlin]
----
// codex-pdf/build.gradle.kts

    .readLines()
    .first { it.startsWith("codex.version") }
    .substringAfter("=")
    .trim()

    implementation("com.cheroliv:codex:$codexVersion")
    implementation("org.apache.pdfbox:pdfbox:3.0.2")
    implementation("org.apache.tika:tika-core:2.9.2")

Bumper la version de tout l'écosystème `codex` = modifier **une seule ligne**
dans `codex/gradle.properties`. Le sous-projet lit la version à la compilation —
pas de version figée dans le code.

==== Chaîne de publication via mavenLocal

[source,bash]
----

==== Ce que ce pattern garantit

*Un seul historique git* ::
Toutes les évolutions de `codex`, `codex-pdf` et `codex-epub` sont dans le même
`git log`. Une PR couvre l'ensemble de l'écosystème. Un tag git versionne
l'ensemble de manière cohérente — le tag correspond à `codex.version`.

*Version unique et sans drift* ::
`codex` dicte la version. Les sous-projets la lisent depuis `gradle.properties`
à chaque build. Il est structurellement impossible qu'un sous-projet soit sur
une version différente du contrat.

*Indépendance de build totale* ::
`./gradlew build` dans `codex-pdf/` ne connaît pas `codex/` au sens Gradle.
Il consomme l'artefact publié dans mavenLocal, exactement comme il consommerait
LangChain4j depuis MavenCentral. Pas de couplage de cycle de build.

*Cohérence avec la gouvernance workspace* ::
Le pattern `publishToMavenLocal` → `implementation("com.cheroliv:...")` est
identique à la chaîne `plantuml-gradle` → `graphify-gradle` → `engine/`. Zéro
exception dans la gouvernance des dépendances.

=== `codex-pdf` — l'implémentation PDF

`codex-pdf` utilise Apache PDFBox pour exploiter les métadonnées typographiques
du PDF — ce que `pdftotext` + `pandoc` ne peuvent pas faire :

[source,kotlin]
----
class PdfDocExtractor : DocExtractor {

    override fun supportsFormat(source: File) =
        source.extension.lowercase() == "pdf"

    override fun extract(source: File): ExtractedDocument {
        val document = PDDocument.load(source)
        val stripper = StructuredPDFTextStripper()
        val blocks = mutableListOf<ExtractedBlock>()

        document.pages.forEachIndexed { pageIdx, page ->
            val textWithFont = stripper.extractTextWithFont(page)
            textWithFont.forEach { span ->
                val blockType = when {
                    span.fontSize > 14f -> BlockType.TITLE
                    span.fontSize > 12f -> BlockType.SUBTITLE
                    span.fontFamily.contains("Mono", ignoreCase = true) -> BlockType.CODE
                    span.fontFamily.contains("Italic", ignoreCase = true) -> BlockType.NOTE
                    else -> BlockType.BODY
                }
                blocks += ExtractedBlock(
                    type = blockType,
                    content = span.text,
                    pageNumber = pageIdx + 1,
                    fontSizePt = span.fontSize,
                    fontFamily = span.fontFamily
                )
            }
        }

        val info = document.documentInformation
        return ExtractedDocument(
            sourcePath = source.absolutePath,
            title = info.title ?: source.nameWithoutExtension,
            authors = listOfNotNull(info.author),
            blocks = blocks,
            metadata = mapOf(
                "pages" to document.numberOfPages.toString(),
                "producer" to (info.producer ?: "unknown")
            )
        )
    }
}

== La Structuration Relationnelle du Livre — JSON LDD, SQL LMD, Cartographie Ontologique

=== La conversation du 7 mai 2026

Ce chapitre capture une conversation tenue le 7 mai 2026 entre cheroliv et
opencode (deepseek-v4-pro). Elle affine la vision du pipeline d'acquisition
documentaire en posant une couche de raffinement supplémentaire :
la **structuration relationnelle du corpus livresque** comme préalable à
toute cartographie ontologique automatisée.

=== Le diagnostic initial

Le plugin `codex-pdf` (chapitre <<codex>>) extrait des blocs typographiques —
`TITLE`, `CODE`, `BODY`, etc. — et les chunk pour le RAG. Mais le chunking
RAG, même sémantique, ne capture pas la structure *relationnelle* d'un livre :
les dépendances entre chapitres, la hiérarchie des concepts introduits, les
références croisées explicites, les patrons de cooccurrence d'entités.

Le RAG répond à "trouve-moi le passage qui ressemble à ma question". Il ne
répond pas à "montre-moi comment les concepts de ce livre s'articulent entre
eux". Or, pour le Planning Expert et le développement par surveillance,
c'est précisément cette articulation qui compte.

=== La thèse : JSON LDD → SQL LMD → ontologie calculée

L'idée-force est de modéliser le livre comme une structure de données
relationnelle :

*Le JSON est le LDD (Langage de Définition des Données)* — il décrit le schéma
canonique du livre : parties, chapitres, sections, entités mentionnées,
références croisées, blocs de code. Ce schéma est le même pour tout livre
technique, même si chaque livre a sa propre topologie interne.

*Le SQL est le LMD (Langage de Manipulation des Données)* — une fois le livre
importé en base relationnelle, on peut le requêter, le joindre, le filtrer,
l'agréger. On passe de "je cherche un passage" à "je comprends le livre".

*L'ontologie est calculée, pas devinée* — les bounded contexts, agrégats et
value objects du domaine émergent du requêtage statistique sur le corpus
structuré. Ce n'est pas une hypothèse a priori : c'est le résultat d'un
`SELECT ... GROUP BY ... HAVING COUNT(*) > N`.

=== Pourquoi AsciiDoc d'abord et pas JSON directement ?

Le flux `PDF → JSON` est une tentation — moins d'étapes, pipeline plus court.
Mais c'est une erreur architecturale pour trois raisons :

. *L'AsciiDoc capte l'intention structurelle de l'auteur* — les niveaux de
titres (`=`, `==`, `===`), les blocs de code (`[source,java]`), les callouts,
les tables. C'est un format qui *sait* représenter la hiérarchie d'un livre.
Le JSON, lui, ne sait pas — il faut déjà avoir décidé du schéma avant de
mapper. L'AsciiDoc est le format intermédiaire qui préserve la structure sans
préjuger du modèle.

. *Auditabilité humaine* — on ouvre le `.adoc`, on voit immédiatement si
l'extraction est fidèle au PDF source. Un JSON de 16 000 lignes de contenu
brut, personne ne le relit intégralement. La boucle de vérification est
cassée si on saute l'étape AsciiDoc.

. *AsciidoctorJ → JSON est déterministe* — une fois le `.adoc` structuré
validé, en extraire le JSON LDD est un simple walk d'AST AsciidoctorJ
(dépendance déjà présente dans `doc-pipeline/build.gradle.kts`). Ce n'est
pas une seconde extraction qui pourrait diverger du PDF source — c'est une
transformation déterministe sur une représentation déjà validée.

=== Ce que le LMD débloque concrètement

==== Exemple sur le livre Applied AI for Enterprise Java Development

Le livre d'Alex Soto Bueno, Markus Eisele et Natale Vinto (O'Reilly, 2025)
couvre LangChain4j aux chapitres 8-9 et LangGraph4j au chapitre 10. Une fois
importé en base, trois requêtes suffisent à cartographier son ontologie :

*Requête 1 — Lignage des patterns (l'ontologie de l'auteur) :*
[source,sql]
----
SELECT c.number AS chapter, s.title AS section, e.name AS entity, e.type
FROM entity e
JOIN section s ON e.section_id = s.id
JOIN chapter c ON s.chapter_id = c.id
WHERE c.number BETWEEN 8 AND 10
ORDER BY c.number, s.order_in_chapter;
----
→ Sortie : `Tool` introduit chap 8 → `Agent` chap 8 advanced → `Routing Agent`
chap 10 → `CheckpointSaver` chap 10. La hiérarchie émerge du séquencement
même des chapitres.

*Requête 2 — Graphe de dépendances inter-chapitres :*
[source,sql]
----
SELECT c1.title AS introduced_in, c2.title AS reused_in, cr.type
FROM cross_reference cr
JOIN chapter c1 ON cr.from_chapter = c1.id
JOIN chapter c2 ON cr.to_chapter = c2.id
WHERE cr.type IN ('prerequisite', 'concept_application', 'pattern_reuse')
ORDER BY c1.number;
----
→ Sortie : `MCP/A2A (chap 4)` est prerequisite de `Routing Agents (chap 10)`,
`Tool Calling (chap 8)` est pattern_reuse dans `Human Interaction (chap 10)`.
Le livre documente lui-même sa propre architecture conceptuelle — le LMD
l'explicite.

*Requête 3 — Bounded contexts candidats par cooccurrence :*
[source,sql]
----
SELECT e1.name, e2.name, COUNT(*) AS cooccurrences
FROM entity_mention e1
JOIN entity_mention e2 ON e1.section_id = e2.section_id AND e1.name < e2.name
GROUP BY e1.name, e2.name
HAVING COUNT(*) > 2
ORDER BY cooccurrences DESC;
----
→ Sortie : `LangGraph4j + State + CheckpointSaver` cooccurrent 12 fois →
bounded context `Orchestration`. `LangChain4j + ChatModel + Prompt + AiServices`
cooccurrent 18 fois → bounded context `Interaction`. Les agrégats ne sont pas
supposés — ils sont *calculés* par densité relationnelle.

==== Généralisation à tout corpus technique

Le même jeu de requêtes s'applique à n'importe quel livre technique importé
en base :
* *Hiérarchie des patterns* → comment l'auteur a intentionnellement séquencé
  ses concepts
* *Graphe de dépendances* → quels chapitres sont prérequis de quels autres
* *Cooccurrence d'entités* → quels concepts forment des clusters naturels =
  candidats bounded contexts

À l'échelle de `office/books-collection/` (plusieurs dizaines de livres),
le LMD permet des requêtes *cross-books* :
* "Quels livres introduisent `LangGraph4j` avant `LangChain4j` ?"
* "Dans quels livres `pgvector` et `Graphify` sont-ils traités dans le même chapitre ?"
* "Quel est le livre qui couvre le plus de patterns du domaine `Orchestration` ?"

=== Enrichissement Croisé — JSON LDD + RAG + Knowledge Graph

==== Le problème résolu

Le flux actuel est linéaire — chaque étape produit un artefact consommé par la
suivante. Or, le même livre existe sous trois représentations qui ne se parlent
pas encore :

[cols="1,3,2", options="header"]
|===
| Représentation | Nature | Outil / Stockage
| **JSON LDD** | Structure déclarative : le squelette du livre (chapitres,
sections, entités, cross-references) | Parsé depuis AsciiDoc, stocké en fichier
| **Chunks RAG** | Matière sémantique : texte découpé par section, vectorisé,
requêtable par similarité cosinus | pgvector + embeddings ONNX AllMiniLmL6V2
| **Knowledge Graph** | Relations exactes entre entités : dépendances,
hiérarchies, cooccurrences | Graphify (graphe navigable, requêtage par prédicat)
|===

Les 4 tâches actuelles (`extractBookStructure`, `asciiDocToJsonLdd`,
`importBookSql`, `deriveOntology`) produisent ces trois représentations
indépendamment. Mais elles ne sont pas jointes. Le JSON LDD ne sait pas
quel chunk RAG correspond à quelle section. Le chunk RAG ne sait pas
quelles entités Graphify sont mentionnées dans son texte. Le nœud Graphify
ne sait pas de quel livre il provient ni dans quel chapitre il apparaît.

C'est un gaspillage de contexte : le vecteur composite assemble RAG + KG +
SQL comme trois composantes séparées, mais le lien entre elles n'est pas
explicite. Le LLM reçoit trois vues du même objet sans savoir qu'elles
décrivent la même chose.

==== La solution : enrichJsonLdd

La cinquième tâche, `enrichJsonLdd`, croise les trois représentations en
ajoutant des pointeurs bidirectionnels dans le JSON LDD. Pour chaque section
du livre, elle :

. *Joint les chunks RAG* par la clé composite `(sourceDocument, pageNumber,
sectionPath)`. Chaque section reçoit la liste des `chunk_id` qui vectorisent
son contenu, plus le score de similarité du chunk le plus proche de la section
mère (le chapitre).
. *Joint les nœuds Graphify* par `entity.name`. Chaque entité mentionnée
dans une section reçoit l'identifiant du nœud correspondant dans le knowledge
graph — qu'il ait été généré par le même livre ou par un autre document du
corpus.
. *Calcule la densité sémantique de la section* — ratio du nombre d'entités
uniques par paragraphe. Les sections à forte densité sont des candidates
naturelles à devenir des bounded contexts.

==== Ce que l'enrichissement croisé débloque concrètement

*Pour le vecteur composite* ::
Au lieu de trois composantes séparées (RAG, KG, SQL), le vecteur composite
devient un graphe unifié. Le LLM reçoit une section avec ses chunks RAG
pré-remontés, ses nœuds Graphify résolus, et ses entités cross-book identifiées.
Il ne "découvre" pas la relation entre `MCP` et `A2A` — il la reçoit déjà
structurellement. La densité d'information par token injecté est maximale.

*Pour le Planning Expert* ::
Quand le Planning Expert décompose une intention, il peut naviguer les trois
couches simultanément :
- Chunks RAG → "quelles sections couvrent le mieux ce besoin ?" (similarité)
- JSON LDD enrichi → "dans quel livre, quel chapitre, quelle séquence ?" (topologie)
- Graphify → "quels concepts sont prérequis de ce concept ?" (relations exactes)

La décomposition n'est plus une recherche en aveugle — c'est une traversée
de graphe dont les arêtes sont déjà calculées.

*Pour le développement par surveillance* ::
Le nœud LangGraph4j qui décide de router vers un outil Gradle peut consulter
le JSON LDD enrichi pour savoir si la tâche demandée est documentée dans le
corpus, par quel livre, avec quelle densité d'entités. Si le corpus contient
une section à densité > 5.0 sur le pattern demandé, le graphe route vers une
exécution assistée. Si la densité est < 1.0, il interrompt et demande
confirmation humaine.

*Pour le LMD cross-book* ::
Une fois plusieurs livres importés et enrichis, le LMD peut répondre à des
requêtes qui combinent les trois représentations :

[source,sql]
----
-- Sections à forte densité d'entités traitant du même concept
-- dans deux livres différents — divergence de définition ?
SELECT b1.title AS book_1, s1.title AS section_1,
       b2.title AS book_2, s2.title AS section_2,
       e.name AS shared_entity,
       s1.entity_density AS density_1,
       s2.entity_density AS density_2
FROM section s1
JOIN book b1 ON s1.book_id = b1.id
JOIN section_entity se1 ON s1.id = se1.section_id
JOIN entity e ON se1.entity_name = e.name
JOIN section_entity se2 ON e.name = se2.entity_name
JOIN section s2 ON se2.section_id = s2.id AND s2.book_id != s1.book_id
JOIN book b2 ON s2.book_id = b2.id
WHERE s1.entity_density > 3.0 AND s2.entity_density > 3.0
ORDER BY e.name, s1.entity_density DESC;
----

→ Sortie : les sections où deux auteurs traitent le même concept avec une
densité comparable — le matériau brut pour l'alignement ontologique
cross-source.

==== La tâche Gradle `enrichJsonLdd`

[source,kotlin]
----
abstract class EnrichJsonLdd : DefaultTask() {

    @get:InputFile
    abstract val jsonLdd: RegularFileProperty

    @get:Input
    abstract val pgvectorUrl: Property<String>

    @get:Input
    abstract val graphifyGraph: RegularFileProperty

    @get:OutputFile
    abstract val enrichedJson: RegularFileProperty

    @TaskAction
    fun enrich() {
        val book = Json.decodeFromString<Book>(jsonLdd.asFile.get().readText())

        val enrichedBook = book.copy(
            parts = book.parts.map { part ->
                part.copy(
                    chapters = part.chapters.map { chapter ->
                        chapter.copy(
                            sections = chapter.sections.map { section ->
                                // 1. Joint RAG chunks
                                val ragChunks = pgvector.query(
                                    filter = "source_document = '${book.isbn}' "
                                        + "AND page_number = ${section.pageNumber} "
                                        + "AND section_path @> '${section.title}'",
                                    topK = 10
                                ).map { RagRef(it.id, it.similarity) }

                                // 2. Joint nœuds Graphify
                                val enrichedEntities = section.entities.map { entity ->
                                    val graphNode = graphify.lookup(entity.name)
                                    entity.copy(
                                        graphNodeId = graphNode?.id,
                                        relatedInGraph = graphNode?.neighbors(2)
                                            ?.map { it.name } ?: emptyList()
                                    )
                                } + graphify.searchCrossBook(section.entities.map { it.name })

                                // 3. Calcule la densité sémantique
                                val entityDensity = section.entities.size.toFloat()
                                    / maxOf(section.content.split("\n\n").size, 1)

                                section.copy(
                                    enrichedEntities = enrichedEntities,
                                    ragChunks = ragChunks,
                                    entityDensity = entityDensity
                                )
                            }
                        )
                    }
                )
            }
        )

        enrichedJson.asFile.get().writeText(Json.encodeToString(enrichedBook))
        logger.lifecycle("[CTX] enrichJsonLdd: ${book.entitiesTotal} entités → "
            + "${enrichedBook.graphNodesResolved} nœuds Graphify résolus, "
            + "${enrichedBook.ragChunksTotal} chunks RAG joints")
    }
}
----

=== La Bibliothèque — Le Sur-Ensemble Systémique Cross-Books

==== Le concept

L'enrichissement croisé traite un livre à la fois : il croise son JSON LDD
avec ses propres chunks RAG et les nœuds Graphify globaux. Mais le vrai
levier, c'est le **cross-book** — la capacité à faire système entre
plusieurs livres pour identifier des relations qui n'existent *dans aucun
livre pris isolément*, mais qui émergent de leur mise en relation.

Une bibliothèque est un **sur-ensemble systémique** qui englobe N livres.
Au sein de ce système, quatre critères de recoupement permettent
d'identifier des relations invisibles au niveau du livre individuel :

[cols="1,3,2", options="header"]
|===
| Critère | Ce qu'il détecte | Source
| **Chunks RAG** | Similarité sémantique cross-books : deux sections de deux
livres différents traitent du même sujet sans mentionner le même terme |
pgvector (cross-book query, même embedding space)
| **Nœuds de connaissance** | Identité conceptuelle : `AggregateRoot` dans le
livre A est le même concept que `Aggregate` dans le livre B | Graphify
(résolution d'entités par voisinage de graphe)
| **Ontologie** | Alignement de bounded contexts : le bounded context
`Orchestration` dérivé du livre A recouvre partiellement le bounded context
`Workflow` dérivé du livre B | SQL LMD (jointure sur entités partagées +
comparaison de densité)
| **Texte / Motifs** | Patterns textuels identiques : deux livres utilisent
la même structure d'explication pour introduire un concept (définition →
exemple → contre-exemple) | Recherche plein texte + regex structurels sur
le contenu des sections
|===

==== Les quatre critères — implémentation

===== Critère 1 : Chunks RAG cross-books

Tous les chunks de tous les livres sont vectorisés dans le même espace
d'embedding (ONNX AllMiniLmL6V2). Une requête cross-book compare les
chunks d'un livre A avec ceux d'un livre B par similarité cosinus :

[source,sql]
----
SELECT a.section_id AS section_a, b.section_id AS section_b,
       a.book_title AS book_a, b.book_title AS book_b,
       1 - (a.embedding <=> b.embedding) AS similarity
FROM chunk_embedding a
CROSS JOIN chunk_embedding b
WHERE a.book_isbn != b.book_isbn
  AND 1 - (a.embedding <=> b.embedding) > 0.85
ORDER BY similarity DESC;
----

Ce qui émerge : deux sections qui ne mentionnent pas les mêmes termes mais
que le modèle juge sémantiquement proches. C'est le "flou productif" du RAG
appliqué au cross-book — il trouve des parentés qu'aucun index plein texte
ne verrait.

===== Critère 2 : Nœuds de connaissance Graphify

Graphify contient des nœuds pour toutes les entités de tous les livres.
La résolution d'identité conceptuelle fonctionne par **voisinage de
graphe** : si deux entités de deux livres différents partagent les mêmes
voisins dans le graphe (mêmes relations, mêmes entités connectées), elles
sont probablement le même concept.

[source,kotlin]
----
fun resolveEntityIdentity(entityA: GraphNode, entityB: GraphNode): Float {
    val neighborsA = entityA.neighbors(2).map { it.name }.toSet()
    val neighborsB = entityB.neighbors(2).map { it.name }.toSet()
    val intersection = neighborsA.intersect(neighborsB).size
    val union = neighborsA.union(neighborsB).size
    return intersection.toFloat() / union.toFloat()  // Jaccard sur voisinage
}
----

Seuil de résolution : Jaccard > 0.6 → les deux entités sont déclarées
équivalentes. La bibliothèque enregistre la relation avec le `graph_path`
(chemin dans le graphe qui a permis la résolution).

===== Critère 3 : Ontologie — alignement des bounded contexts

Chaque livre produit ses propres bounded contexts candidats via
`deriveOntology`. La bibliothèque les aligne par **jointure sur entités
partagées** :

[source,kotlin]
----
data class BoundedContextAlignment(
    val bcBookA: BoundedContext,
    val bcBookB: BoundedContext,
    val sharedEntities: List<String>,
    val overlapRatio: Float,         // |A ∩ B| / min(|A|, |B|)
    val densityCorrelation: Float     // corrélation des densités
)

fun alignBoundedContexts(bookA: List<BoundedContext>,
                         bookB: List<BoundedContext>): List<BoundedContextAlignment> {
    return bookA.flatMap { bcA ->
        bookB.mapNotNull { bcB ->
            val shared = bcA.entities.intersect(bcB.entities.toSet())
            if (shared.size < 2) return@mapNotNull null // minimum 2 entités partagées
            val overlap = shared.size.toFloat() / minOf(bcA.entities.size, bcB.entities.size)
            if (overlap < 0.4) return@mapNotNull null    // seuil de recouvrement
            BoundedContextAlignment(bcA, bcB, shared.toList(), overlap,
                pearsonCorrelation(bcA.entityDensities, bcB.entityDensities))
        }
    }.sortedByDescending { it.overlapRatio }
}
----

L'alignement ne dit pas "ces deux bounded contexts sont identiques" — il
dit "ces deux bounded contexts partagent N entités, avec un ratio de
recouvrement R, et une corrélation de densité D". C'est à l'humain (ou au
Planning Expert) de trancher si l'alignement est pertinent.

===== Critère 4 : Texte / Motifs — patterns structurels

Au-delà du contenu sémantique, les livres techniques suivent des **patterns
d'exposition** récurrents : définition formelle → exemple → contre-exemple,
ou problème → patterns connus → solution émergente, ou concept A introduit
au chapitre N → réutilisé au chapitre N+3.

Ces motifs sont détectables par des **requêtes structurelles** sur le JSON
LDD enrichi :

[source,kotlin]
----
enum class ExpositionPattern {
    DEFINE_THEN_EXEMPLIFY,     // "X is..." suivi d'un bloc de code
    PROBLEM_PATTERNS_SOLUTION, // "The problem..." → patterns → solution
    INTRODUCE_THEN_REUSE,      // concept introduit chap N, réutilisé chap N+K
    COMPARE_CONTRAST,          // "Unlike X, Y..." — comparaison explicite
    HISTORICAL_MOTIVATION      // "Before X, developers used Y..."
}

fun detectExpositionPatterns(sections: List<EnrichedSection>): Map<ExpositionPattern, Int> {
    return mapOf(
        DEFINE_THEN_EXEMPLIFY to sections.count {
            it.content.contains(" is ", ignoreCase = true)
                && it.codeBlocks.isNotEmpty()
                && it.codeBlocks.first().charOffset < it.content.indexOf(" is ") + 200
        },
        INTRODUCE_THEN_REUSE to countCrossReferences(sections) { ref ->
            ref.from.chapterNumber < ref.to.chapterNumber
                && ref.type == "pattern_reuse"
        }
    )
}
----

Deux livres qui utilisent le même pattern d'exposition pour le même concept
sont probablement dans une relation de filiation intellectuelle (l'un a
inspiré l'autre, ou les deux puisent à la même source). La bibliothèque
enregistre cette relation comme `text_pattern: both_define_with_example`.

==== Les relations qui émergent

Sur les quatre critères, la bibliothèque calcule un **score composite de
relation cross-book** :

[source,kotlin]
----
data class CrossBookRelation(
    val from: SectionRef,
    val to: SectionRef,
    val criteriaScores: Map<Criteria, Float>,  // chunk: 0.89, node: 0.92, ontology: 0.0, text: 0.75
    val compositeScore: Float,                  // moyenne pondérée des scores présents
    val confidence: Float,                       // ratio critères activés / 4
    val relationType: RelationType               // equivalence, complement, contradiction, inspiration
)

enum class Criteria { CHUNK, KNOWLEDGE_NODE, ONTOLOGY, TEXT_PATTERN }

enum class RelationType {
    CONCEPT_EQUIVALENCE,    // même concept, mots différents → à fusionner
    COMPLEMENT,             // concept A dans livre 1, concept B dans livre 2, A → B
    CONTRADICTION,          // définitions incompatibles → à trancher
    INSPIRATION,            // même pattern d'exposition, même structure → filiation
    NOVELTY_ISOLATED         // concept présent dans un seul livre → lacune cross-corpus
}
----

Le `compositeScore` pondère également les quatre critères : le chunk RAG
(0.3) est le plus "flou" mais le plus sensible, le nœud Graphify (0.3) est
le plus précis, l'ontologie (0.2) capture la structure métier, le motif
textuel (0.2) capture la forme d'exposition.

==== La tâche Gradle `assembleLibrary`

`assembleLibrary` est la tâche qui construit le JSON LDD de la bibliothèque
à partir de tous les JSON LDD enrichis du corpus. Elle est incrémentale :
si un seul livre est modifié, seules les relations qui l'impliquent sont
recalculées.

[source]
----
assembleLibrary (entrée : tous les JSON LDD enrichis de office/data-engineering/doc-pipeline/json-ldd/)

    │

==== Ce que la bibliothèque systémique permet

*Pour le Planning Expert* ::
Quand le Planning Expert reçoit une intention, il ne cherche pas "dans quel
livre ce concept est-il documenté ?" — il interroge la bibliothèque
directement. La bibliothèque lui répond avec les sections pertinentes de
*plusieurs* livres, déjà reliées entre elles par les quatre critères. Le
plan généré n'est pas ancré sur un seul ouvrage — il est synthétisé depuis
le corpus entier.

*Pour la curation du corpus* ::
Les entités isolées (présentes dans un seul livre) sont des candidates à
l'acquisition : "ce concept n'est couvert que par un seul livre, faut-il
acquérir d'autres sources ?" Inversement, les concepts canoniques
(présents dans ≥ 3 livres) sont des candidats à l'export comme documentation
de référence.

*Pour la résolution de contradictions* ::
Quand deux livres définissent le même concept de manière incompatible
(relation `CONTRADICTION`), la bibliothèque le signale. Le Planning Expert
peut alors router vers un nœud LangGraph4j d'interruption humaine :
"L'auteur A définit X comme Y, l'auteur B définit X comme Z. Laquelle
adopter ?"

*Pour le vecteur composite* ::
La bibliothèque devient la quatrième composante du vecteur composite, aux
côtés de RAG, KG et SQL. Elle fournit la **vue cross-book** — la capacité
à situer un concept dans le paysage documentaire complet, pas seulement
dans le livre qui le mentionne.

==== Le principe de la bibliothèque

> La bibliothèque n'est pas une collection de livres. C'est un système dont
> les livres sont les composants. Les relations entre livres ne sont pas
> des métadonnées ajoutées après coup — elles sont calculées sur quatre
> critères indépendants (similarité sémantique, identité conceptuelle,
> alignement ontologique, patterns textuels). Ce qui émerge de ce calcul,
> c'est une carte du savoir que le corpus *contient déjà* mais qu'aucun
> livre pris isolément ne peut exprimer.

==== Implémentation concrète

`assembleLibrary` consomme directement les artefacts existants :

- *pgvector* est déjà en place (embeddings ONNX, requêtage vectoriel)
- *Graphify* est déjà en place (109 tests, 380/380 PASS, graphe navigable)
- *SQL LMD* est déjà en place (PostgreSQL 16.3, `importBookSql` génère le DDL)
- *JSON LDD enrichis* sont produits par `enrichJsonLdd`

Aucune nouvelle infrastructure. Aucune nouvelle dépendance. La bibliothèque
est une **tâche Gradle de jointure** qui lit les artefacts existants et
calcule les relations. Le JSON produit est lui-même importable en SQL pour
le LMD cross-book.

Le seul prérequis : avoir au moins deux livres importés et enrichis dans
le pipeline. Le livre Applied AI est le premier. Le deuxième pourrait être
Implementing Domain-Driven Design (Vernon) ou un autre PDF technique déjà
présent dans `office/books-collection/`.

=== La Taxonomie de Bloom — Le Niveau d'Appréhension par Concept

==== La cohérence découverte

Les livres techniques ne sont pas des catalogues de concepts — ce sont des
**trajectoires d'apprentissage**. Chaque concept y apparaît à un certain
niveau d'appréhension : défini (Remember), illustré (Understand), mis en œuvre
(Apply), comparé (Analyze), critiqué (Evaluate), recombiné (Create).

Cette progression est la taxonomie de Bloom, implicite dans la structure du
livre, jamais nommée. Le pipeline peut la **rendre explicite** en étiquetant
chaque section avec son niveau Bloom, et en cartographiant la progression des
concepts à travers les chapitres et les livres.

La connexion avec l'architecture existante est naturelle : les quatre critères
de la bibliothèque gagnent une cinquième dimension — le niveau d'appréhension
Bloom — qui qualifie non pas *quoi* mais *à quelle profondeur* un concept est
traité.

==== Les six niveaux — mapping automatique

[cols="1,2,3,4", options="header"]
|===
| Niveau Bloom | Verbes marqueurs | Signature | Exemple Applied AI
| **Remember**
_Niveau 1_ | define, list, recall, name, identify | Définition sans explication
| « LangChain4j is a Java implementation inspired by LangChain » (chap 8)
| **Understand**
_Niveau 2_ | explain, describe, summarize, interpret | Paragraphe explicatif
suivant la définition | « It helps developers build applications that
integrate with LLMs » (chap 8, l.6913)
| **Apply**
_Niveau 3_ | implement, demonstrate, execute, use | Bloc de code dans la
section | « Add the following dependency: langchain4j-open-ai... » (chap 8,
l.6943)
| **Analyze**
_Niveau 4_ | compare, contrast, categorize, examine | Comparaison explicite
ou tableau comparatif | « In contrast, using Jlama with LangChain4j... »
(chap 7, l.5202)
| **Evaluate**
_Niveau 5_ | assess, critique, justify, recommend | Verdict qualitatif,
discussion de limites | « We recommend using LangChain4j to future-proof
your Java AI solutions » (chap 12, l.14345)
| **Create**
_Niveau 6_ | design, build, compose, generate | Assemblage de patterns en
solution nouvelle | « Advanced RAG schema with self-reflection » (chap 10,
l.12770) — RAG + KG + agent loop unifiés
|===

===== Algorithme de classification Bloom

[source,kotlin]
----
enum class BloomLevel(val verbs: List<String>, val order: Int) {
    REMEMBER(listOf("define", "is a", "refers to", "named"), 1),
    UNDERSTAND(listOf("explain", "this means", "in other words",
                       "for example"), 2),
    APPLY(listOf("implement", "add the", "run", "configure"), 3),
    ANALYZE(listOf("compare", "unlike", "trade-off", "differs"), 4),
    EVALUATE(listOf("recommend", "best practice", "avoid",
                     "limitation", "works well", "fails when"), 5),
    CREATE(listOf("putting it all together", "final architecture",
                   "end-to-end", "combining", "design a"), 6);

    fun match(text: String): Float {
        val hits = verbs.count { text.lowercase().contains(it) }
        return hits.toFloat() / verbs.size
    }
}

fun classifySectionBloom(section: EnrichedSection): Map<BloomLevel, Float> {
    return BloomLevel.entries.associateWith { it.match(section.content) }
}
----

Le score n'est pas binaire — une section peut avoir `APPLY=0.7` et
`UNDERSTAND=0.4`. La classification retient le niveau dominant, les scores
partiels sont conservés pour les requêtes croisées.

> propriété navigable du graphe de connaissance, requêtable en SQL,
> filtrable dans le vecteur composite.
>
> Bloom n'est pas une couche externe — c'est une propriété émergente du
> corpus que le pipeline rend explicite. Le livre Applied AI *est déjà*
> une progression Bloom. Le pipeline la nomme.

=== Les Couches d'Augmentation Manquantes — Code, Delta, Erreur, Dépendances

Le pipeline extrait et croise la connaissance livresque. Mais l'agent
LangChain4j opère dans un environnement vivant — du code qui évolue, des builds
qui échouent, des dépendances dont la documentation n'a jamais été acquise.
Quatre couches d'augmentation manquent pour rendre l'agent pleinement conscient
de son environnement d'exécution.

==== Couche 1 — Code-to-Corpus Traceability

===== Le problème

Aujourd'hui, le pipeline relie les livres entre eux (bibliothèque, Bloom
cross-book). Mais le code source n'est pas relié au corpus. `AuthService.kt`
implémente un `CircuitBreaker` — l'agent ne sait *pas* que ce pattern est
documenté, où, et à quel niveau Bloom.

===== La solution

Une annotation `@ConceptSource` et/ou un fingerprint de code qui associe
chaque fichier source aux sections du corpus qui documentent ce qu'il
implémente :

[source,kotlin]
----
@ConceptSource(
    books = [
        BookRef(isbn = "978-1-098-17675-4", chapter = 12,
                section = "Guardrails and Circuit Breaking", bloom = EVALUATE),
        BookRef(isbn = "978-1-492-07698-8", chapter = 5,
                section = "Resilience Patterns", bloom = APPLY)
    ],
    patterns = ["CircuitBreaker", "Retry", "Fallback"],
    confidence = 0.88  // score du matching automatique
)
@Service
class PaymentService(
    @CircuitBreaker(name = "paymentGateway") val gateway: PaymentGateway
)
----

*Matching automatique* : pour chaque fichier Kotlin/Java, le pipeline détecte
les imports, annotations, et noms de classe → recherche dans les entités du
corpus → calcule un score de confiance → associe les sections pertinentes.

===== Ce que ça débloque

- L'agent reçoit le fichier avec son **ancrage documentaire** pré-calculé
- « Ce `CircuitBreaker` est documenté dans Applied AI (EVALUATE) + Resilience4j docs (APPLY) »
- Le Planning Expert peut auditer : « 12 patterns identifiés dans le code,
  8 sont documentés dans le corpus, 4 sont orphelins → acquérir de la doc »

==== Couche 2 — Change Delta Awareness

===== Le problème

L'agent reçoit le contexte du workspace, mais pas ce qui a *changé* depuis
la dernière exécution. Il traite l'état comme statique alors que le code
évolue entre deux invocations.

===== La solution

Une composante `delta` du vecteur composite construite depuis `git diff` :

[source,kotlin]
----
data class WorkspaceDelta(
    val sinceCommit: String,
    val filesAdded: List<String>,       // nouveaux fichiers
    val filesModified: List<String>,    // fichiers modifiés
    val filesDeleted: List<String>,     // fichiers supprimés
    val modulesTouched: Set<String>,    // modules impactés (auth, payment...)
    val diffSummary: String,            // git diff --stat résumé
    val relatedSections: List<SectionRef>  // sections du corpus liées aux fichiers modifiés
)

fun computeDelta(workspace: File, since: String = "HEAD~1"): WorkspaceDelta {
    val diff = Git(workspace).diff().setOldVersion(since).call()
    val filesAdded = diff.addedFiles
    val filesModified = diff.modifiedFiles
    val modulesTouched = (filesModified + filesAdded)
        .mapNotNull { it.moduleName() }
        .toSet()

    return WorkspaceDelta(
        sinceCommit = since,
        filesAdded = filesAdded,
        filesModified = filesModified,
        filesDeleted = diff.deletedFiles,
        modulesTouched = modulesTouched,
        diffSummary = Git(workspace).diff().setStat().call(),
        relatedSections = resolveSectionsForFiles(filesModified)
    )
}
----

- « Depuis le dernier build : 3 fichiers modifiés, 1 nouveau, module auth
  refactoré, module payment intact »
- Le Planning Expert priorise ses EPICs sur les modules touchés
- Le développement par surveillance détecte une modification dans une zone
  critique → interruption humaine obligatoire
- `git log --since=yesterday` → historique des décisions intégré au contexte

==== Couche 3 — Error → Anti-Pattern Mapping

===== Le problème

Les stacktraces et échecs de build ne sont pas mis en relation avec le
corpus. L'agent signale l'erreur mais ne la *diagnostique* pas.

===== La solution

Une composante `failure_context` qui joint les erreurs avec les sections
du corpus documentant le pattern fautif :

[source,kotlin]
----
data class FailureDiagnosis(
    val error: Throwable,
    val location: SourceLocation,       // fichier:ligne
    val matchedAntiPatterns: List<AntiPatternMatch>,
    val suggestedCorrections: List<SectionRef>,
    val bloomGap: BloomLevel?           // le code est à APPLY ou CREATE mais
                                        // l'erreur vient d'un concept mal COMPRIS
)

data class AntiPatternMatch(
    val section: SectionRef,
    val patternName: String,
    val confidence: Float,
    val excerpt: String                 // extrait du corpus qui décrit l'anti-pattern
)

fun diagnoseFailure(stacktrace: String, corpus: Library): FailureDiagnosis {
    val error = parseStacktrace(stacktrace)
    val location = error.topFrame

    // Matching par type d'exception + contexte de code
    val matches = corpus.sections
        .filter { section ->
            section.bloom?.dominant in listOf(EVALUATE, CREATE)
                && section.antiPatterns.any { it.matches(error) }
        }
        .map { section ->
            AntiPatternMatch(
                section = section.ref,
                patternName = section.antiPatterns.first { it.matches(error) }.name,
                confidence = calculateMatchConfidence(error, section),
                excerpt = section.content.substringAroundMatch(error)
            )
        }
        .sortedByDescending { it.confidence }

    return FailureDiagnosis(error, location, matches,
        matches.firstOrNull()?.section?.ref,
        // Le concept sous-jacent est-il au bon niveau Bloom dans le corpus ?
        detectBloomGap(error, location))
}
----

==== Couche 4 — Dépendances : Extraction de la Documentation des Bibliothèques

===== Le problème

Ton code dépend de bibliothèques (LangChain4j, Spring Boot, PDFBox, Graphify...)
dont la documentation officielle n'a jamais été acquise par le pipeline. L'agent
connaît le corpus livresque mais ignore la documentation de référence des
dépendances que le code *utilise effectivement*.

===== La solution

La tâche `acquireDependencyDocs` parse les `build.gradle.kts` de tous les
projets du workspace, identifie les dépendances, localise leur documentation
(javadoc, site officiel, README GitHub, guides) et l'ingère dans le pipeline :

fun resolveDocUrls(dep: GradleDependency): List<URL> {
    return buildList {
        // 1. Javadoc officiel
        add(URL("https://javadoc.io/doc/${dep.group}/${dep.artifact}/${dep.version}"))
        // 2. Site du projet (détecté depuis le POM Maven Central)

        pom.scm?.url?.let { add(URL(it)) }
        // 3. GitHub README.md
        val githubUrl = pom.scm?.url?.toGitHubUrl()
        githubUrl?.let { add(URL("$it/blob/main/README.md")) }
        // 4. Guides / Reference docs
        githubUrl?.let { add(URL("$it/tree/main/docs")) }
        // 5. GitHub Wiki
        githubUrl?.let { add(URL("$it/wiki")) }
    }
}

. Parse tous les `build.gradle.kts` du workspace → liste unique des dépendances

==== Ce que les couches Code apportent à la gouvernance agent

L'audit de code devient un processus continu et automatisé :

- *`annotateCodeSources`* scanne le code, détecte les patterns, les ancre dans le

- *`acquireDependencyDocs`* ingère la documentation des bibliothèques → les API docs

Ces quatre tâches ferment la boucle : le pipeline ne se contente plus d'extraire
la connaissance des livres — il **l'injecte dans le code** et **apprend du code**
en retour. Chaque erreur enrichit le corpus. Chaque modification met à jour le
contexte. Chaque dépendance documentée élargit le champ de l'agent.

==== Le principe des couches Code

> Le vecteur composite ne doit pas seulement décrire ce que le corpus contient.
> Il doit décrire ce que le code *fait*, ce qui a *changé*, ce qui a *échoué*,
> et ce que les *dépendances* documentent. Ces quatre couches transforment l'agent
> LangChain4j d'un lecteur de corpus en un **observateur du système vivant**.
>
> L'agent ne reçoit pas un contexte statique — il reçoit une photographie de
> l'état du système à l'instant T, ancrée dans le corpus, annotée par Bloom,
> informée par le delta, et enrichie par la documentation des bibliothèques
> que le code consomme.

==== Propositions additionnelles de mise en contact

Au-delà des six couches documentées, voici les dimensions qui pourraient
encore augmenter le contexte :

*7. Historique des sessions agent*
Toutes les sessions opencode (`.agents/sessions/NNN-*.adoc`) sont elles-mêmes

*9. Pull Request / Code Review Context*
Les PRs ouvertes, les commentaires de review, les approbations/blocages — tout
cela est du contexte décisionnel. « La PR #23 sur le module auth est bloquée

besoin d'aide et où le code est en friction.

*10. Release Notes / Changelogs des dépendances*

`@Tool` 14 fois → plan de migration généré. »

*11. StackOverflow / GitHub Issues / Discussions*
Les questions et réponses publiques sur les bibliothèques que tu utilises sont
un corpus vivant. « 3 issues GitHub ouvertes sur `langgraph4j:checkpoint-saver`.
L'issue #142 décrit exactement ton problème de sérialisation. Workaround posté
il y a 2 jours. »

*12. Ontologie de la Gouvernance Agent elle-même*
Les fichiers `.agents/` sont déjà parsés par `relocalizeAgents`. Mais leur
*structure* peut être modélisée en ontologie. « La règle 2bis (filtre
épistémique) est violée dans 3 sessions récentes. Le pattern de violation
est : mention d'un client dans une section taguée Niveau 3. » L'agent audite
sa propre gouvernance.

=== Les Outils de Raisonnement — La Couche de Consommation Intelligente

==== Le diagnostic

Le pipeline acquiert et structure la connaissance. Mais il lui manque la
couche qui transforme cette connaissance en **inférences** — en nouvelles
conclusions produites par raisonnement, pas par simple recherche.

Un système qui sait *trouver* une information n'est pas un système qui sait
*penser avec*. Les quatre critères de la bibliothèque et les six couches du
vecteur composite répondent à des requêtes. Ils ne raisonnent pas.

Les vingt-et-un outils de raisonnement ci-dessous constituent la **couche de
consommation intelligente** du pipeline. Certains sont directement
algorithmisables (opérations SQL ou graphe sur les données structurées).
D'autres sont du prompt engineering (le LLM reçoit un prompt structuré avec
le corpus et une instruction de raisonnement). Tous partagent le même contrat :
une tâche Gradle typée, un corpus en entrée, une inférence en sortie.

==== Cartographie des outils par implémentabilité

Chaque outil est classé selon qu'il peut être codé en pur Kotlin/SQL/Graphe
(ALGO) ou qu'il nécessite un appel LLM avec un prompt structuré (PROMPT), ou
une combinaison des deux (HYBRIDE).

==== Architecture de consommation : la tâche `reason`

Plutôt que 21 tâches Gradle distinctes, une tâche unifiée `reason` qui
accepte un paramètre `-Pmode` et route vers la bonne stratégie :

[source,bash]
----
# Algorithmique pur — pas d'appel LLM
./gradlew reason -Pmode=compare -PentityA=Agent -PentityB=Tool
./gradlew reason -Pmode=deduct -Ppremises="Agent HAS State, Routing IS Agent"

# Hybride — pré-processing ALGO + conclusion LLM
./gradlew reason -Pmode=abduct -Pobservation="PaymentService:3x@CircuitBreaker,0 test"
./gradlew reason -Pmode=causal -Perror="NullPointerException:PaymentService.java:42"

# Prompt — LLM avec corpus complet
./gradlew reason -Pmode=dialectic -Pconcept="Agent Architecture"
./gradlew reason -Pmode=analogy -Psource="LangGraph4j:Agent" -Ptarget="Spring:Bean"
./gradlew reason -Pmode=systemsThinking -Pscope=workspace
----

===== Implémentation du routeur `reason`

[source,kotlin]
----
abstract class ReasonTask : DefaultTask() {

    @get:Input
    abstract val mode: Property<String>         // compare, deduct, analogize...

    @get:Input
    abstract val entityA: Property<String>

    @get:Input
    abstract val entityB: Property<String>

    @get:Optional
    @get:Input
    abstract val observation: Property<String>

    @get:OutputFile
    abstract val inferenceFile: RegularFileProperty

    @TaskAction
    fun reason() {
        val result = when (mode.get()) {
            "compare" -> compareEntities(entityA.get(), entityB.get())
            "classify" -> classifyEntities(entityA.get())
            "decompose" -> decomposeEntity(entityA.get())
            "infer" -> inferFromGraph(entityA.get(), entityB.get())
            "deduct" -> deductFromRules(entityA.get())
            "distinguish" -> distinguishConcepts(entityA.get(), entityB.get())
            "proportion" -> computeProportion(entityA.get(), entityB.get())

            // Hybrides : ALGO structure → LLM conclusion
            "syllogism", "induct", "abduct", "causal",
            "absurd", "refute" -> hybridReason(mode.get(), entityA.get(),
                entityB.get(), observation.getOrElse(""))

            // Purement prompt : corpus + instruction LLM
            "dialectic", "analogy", "thoughtExperiment",
            "hypotheticoDeductive", "systemsThinking",
            "conceptualize" -> promptReason(mode.get(), entityA.get(),
                entityB.get(), observation.getOrElse(""))
        }

        inferenceFile.asFile.get().writeText(
            StdoutFormatter.result(result)
        )
    }
}
----

==== Ce que les outils de raisonnement apportent

*Pour le Planning Expert* ::
Au lieu de « décomposer l'intention en User Stories », le Planning Expert
peut maintenant raisonner *sur* la décomposition :
- `reason -Pmode=decompose -Pentity="Capsule Video Feature"` → arborescence
  des sous-tâches avec prérequis Bloom
- `reason -Pmode=deduct -Ppremises="EPIC-1 DEPENDS_ON CD-2, CDX-2 BLOCKS EPIC-1"`
  → alerte de blocage avant exécution

*Pour la Bibliothèque cross-book* ::
Les 5 critères de la bibliothèque sont des opérations de comparaison et
d'induction. Les outils de raisonnement les généralisent :
- `reason -Pmode=analogy -Psource="Agent Architecture:Applied AI" -Ptarget="Microservices:Implementing DDD"` → « Les deux architectures sont
  isomorphes : l'agent correspond au service, le tool au client HTTP, le
  state au domain model »
- `reason -Pmode=generalize -Pentities="Agent,Tool,State,Graph"` → abstraction
  « Système Agentique = Graphe + Acteurs + Contrats + État »

*Pour le développement par surveillance* ::
Le nœud LangGraph4j qui supervise peut invoquer `reason` comme un outil :
- Si le build échoue : `reason -Pmode=abduct -Pobservation="Compilation error:AuthService.kt"`
  → hypothèse la plus probable
- Si l'agent hésite entre deux décisions : `reason -Pmode=dialectic -Pconcept="Prochaine action"`
  → thèse/antithèse/synthèse

*Pour la génération de formations* ::
- `reason -Pmode=conceptualize -Pentities="CircuitBreaker,Retry,Fallback"`
  → concept abstrait « Resilience Pattern »
- `reason -Pmode=proportion -PentityA="Applied AI" -PentityB="Implementing DDD"`
  → ratio de couverture par niveau Bloom → quel livre pour quel public

==== Le principe des outils de raisonnement

> Le pipeline ne doit pas seulement *stocker* la connaissance — il doit
> savoir *raisonner avec*. Les vingt-et-un outils de raisonnement sont la
> couche de consommation intelligente : une tâche Gradle `reason` qui,
> selon le mode, exécute une opération algorithmique sur les données
> structurées, un raisonnement hybride (ALGO + LLM), ou un pur prompt
> structuré sur le corpus.
>
> La séparation ALGO / HYBRIDE / PROMPT n'est pas une hiérarchie de valeur —
> c'est une cartographie d'implémentabilité. Un syllogisme codé en règles
> Kotlin est plus rapide et déterministe. Une analogie par LLM capture du
> sens qu'aucun algorithme ne peut extraire. Le pipeline combine les deux
> régimes dans la même interface unifiée : `./gradlew reason -Pmode=...`.

==== Positionnement dans l'architecture

Les outils de raisonnement ne sont pas une couche supplémentaire du vecteur
composite — ils sont une **couche transverse** qui peut interroger toutes les
autres :

[source]
----
./gradlew reason -Pmode=... -Pcontext=rag|graph|sql|library|bloom|code|delta|error|deps|all
----

- `-Pcontext=rag` → le raisonnement s'appuie sur les chunks RAG uniquement
- `-Pcontext=all` → le raisonnement reçoit le vecteur composite complet
  (RAG + KG + SQL + Bibliothèque + Bloom + Code + Delta + Erreurs + Dépendances)

C'est la différence entre « comparer deux entités » (ALGO, contexte SQL)
et « faire une analogie systémique entre deux livres » (PROMPT, contexte all).
La même tâche, le même contrat, un niveau de contexte paramétrable.

=== Le LLM Trainer — L'Expert Qui Fabrique les Experts

==== La cohérence découverte

Le pipeline acquiert la connaissance. La bibliothèque la croise. Bloom la
qualifie. Les outils de raisonnement la consomment. Mais **qui transforme
cette connaissance en expert déployable** ?

Aucun acteur existant n'est responsable de la *naissance* des experts
spécialisés. Les experts CDA (Kotlin/Gradle/JHipster) et FPA
(pédagogie/Qualiopi/Bloom) sont planifiés dans les roadmaps, mais personne
n'a le rôle de les faire exister.

Le LLM Trainer est cet acteur. Son travail : sur fourniture d'une spécification
d'expert, collecter et construire le dataset, le distiller, fine-tuner le
modèle, valider ses compétences, et l'intégrer dans l'écosystème d'agents.

Ce qui rend son recrutement immédiatement pertinent : son travail est déjà
amorcé par les couches existantes du pipeline.

==== Les trois piliers déjà construits pour lui

*Pilier 1 — Le dataset est déjà structuré* ::
Le JSON LDD enrichi (chapitres, sections, entités annotées Bloom, chunks RAG,
nœuds Graphify) est le matériau brut de tout dataset d'entraînement. Le
LLM Trainer n'a pas à extraire — il a à **filtrer et composer**.

*Pilier 2 — Les spécifications d'experts sont déjà formalisées* ::
`PLANNING_EXPERT_ONTOLOGY.adoc` définit le cahier des charges d'un expert :
intention structurée → EPICs → User Stories → tâches Gradle. Le LLM Trainer
reçoit une spec au format connu et sait quelles compétences l'expert doit
maîtriser.

*Pilier 3 — L'évaluation est déjà couverte* ::
`classifyBloom` mesure la profondeur de couverture d'un concept. La
bibliothèque mesure la couverture cross-books. L'audit de couverture
(`auditAugmentationCoverage`) mesure les lacunes. Le LLM Trainer utilise
ces mêmes métriques pour évaluer si le modèle fine-tuné a bien assimilé
le domaine.

==== Les quatre phases du LLM Trainer

===== Phase 1 — Collecte du dataset

À partir d'une spec d'expert (ex: « Expert CDA : Kotlin, Gradle, JHipster,
Spring Boot, hexagonal architecture »), le LLM Trainer lance
`assembleTrainingDataset` :

[source,kotlin]
----
data class ExpertSpec(
    val name: String,                    // "CDA Expert"
    val domains: List<String>,           // ["kotlin", "gradle", "spring-boot"]
    val bloomTarget: BloomLevel,         // CREATE — l'expert doit pouvoir générer
    val boundedContexts: List<String>,   // bounded contexts à couvrir
    val excludedPatterns: List<String>,  // anti-patterns à éviter
    val referenceBooks: List<String>     // ISBNs des livres de référence
)

data class TrainingDataset(
    val expertSpec: ExpertSpec,
    val sections: List<EnrichedSection>,     // sections du corpus filtrées
    val codeExamples: List<CodeBlock>,       // blocs de code extraits
    val qaPairs: List<QuestionAnswerPair>,   // paires Q/R générées
    val patterns: List<DesignPattern>,       // patterns documentés
    val graphSubgraph: KnowledgeGraph,       // sous-graphe Graphify du domaine
    val coverage: BloomCoverage,             // couverture Bloom par concept
    val split: DatasetSplit                  // train/val/test
)

data class DatasetSplit(
    val train: Float,   // 0.7
    val validation: Float,  // 0.15
    val test: Float     // 0.15
)
----

`assembleTrainingDataset` interroge toutes les couches du pipeline :
. *Corpus livresque* → sections qui mentionnent les entités du domaine,
  filtrées par `bloom >= bloomTarget`
. *Dépendances* → documentation des bibliothèques listées dans la spec
. *Code source* → fichiers Kotlin/Java annotés `@ConceptSource` dans le
  domaine cible, plus les fichiers du workspace qui implémentent ces patterns
. *Knowledge Graph* → extrait le sous-graphe des entités du domaine avec
  leurs relations (imports, dépendances, hiérarchie)
. *Paires Q/R* → générées automatiquement depuis les sections :
  titre = question, contenu = réponse, ou via `reason -Pmode=abduct`

===== Phase 2 — Distillation du dataset

Le dataset brut est trop volumineux pour un fine-tuning efficace. La
distillation réduit le bruit et maximise la densité d'information :

[source,kotlin]
----
data class DistillationConfig(
    val maxTokens: Int = 10_000_000,      // budget tokens total
    val bloomPriority: List<BloomLevel>    // priorité décroissante
        = listOf(CREATE, EVALUATE, ANALYZE, APPLY, UNDERSTAND, REMEMBER),
    val deduplicationThreshold: Float = 0.92, // similarité max entre exemples
    val balanceByConcept: Boolean = true,     // équilibrer les concepts
    val minExamplesPerConcept: Int = 5
)

fun distill(dataset: TrainingDataset,
            config: DistillationConfig): DistilledDataset {
    // 1. Trier par priorité Bloom → prioriser les niveaux hauts
    val sorted = dataset.sections.sortedByDescending {
        config.bloomPriority.indexOf(it.bloom!!.dominant)
    }

    // 2. Dédupliquer → supprimer les exemples trop similaires
    val deduplicated = deduplicateByEmbedding(sorted, config.deduplicationThreshold)

    // 3. Équilibrer → au moins N exemples par concept
    val balanced = balanceByConcept(deduplicated, config.minExamplesPerConcept)

    // 4. Tronquer au budget tokens
    val finalSections = balanced.take(config.maxTokens)

    // 5. Générer les prompts d'entraînement formatés
    val trainingPrompts = finalSections.map { section ->
        TrainingPrompt(
            system = buildSystemPrompt(dataset.expertSpec, section.bloom!!.dominant),
            user = section.content,
            assistant = buildAssistantResponse(section)
        )
    }

    return DistilledDataset(
        prompts = trainingPrompts,
        stats = DistillationStats(
            originalSections = dataset.sections.size,
            distilledSections = finalSections.size,
            compressionRatio = dataset.sections.size.toFloat() / finalSections.size,
            bloomDistribution = bloomDistribution(finalSections),
            conceptsCovered = conceptsIn(finalSections).size
        )
    )
}
----

===== Phase 3 — Fine-tuning

La distillation produit un dataset au format standard (JSON Lines, format
conversationnel). Le LLM Trainer orchestre le fine-tuning :

[source]
----
assembleTrainingDataset  →  distill  →  fine-tune  →  validate  →  deploy
       │                       │           │            │           │
       ▼                       ▼           ▼            ▼           ▼
  dataset.json            distilled    checkpoint    benchmark   Maven Central
  (office/data-          .jsonl       .gguf          results     ou Ollama
   engineering/llm/)                                (Bloom +     local
                                                     coverage)
----

Le fine-tuning lui-même est délégué à des outils externes (Unsloth, Axolotl,
MLX, ou un service cloud), mais le LLM Trainer est responsable de :
- La qualité du dataset en entrée
- La configuration d'entraînement (LoRA rank, learning rate, epochs)
- La validation post-entraînement (benchmark Bloom, coverage, régression)
- La décision go/no-go : déployer ou itérer

===== Phase 4 — Intégration dans l'écosystème

Une fois fine-tuné et validé, l'expert est intégré :

. *Publication* — modèle publié en `.gguf` dans `office/data-engineering/llm/`
  ou sur Ollama/HuggingFace
. *Enregistrement* — le JSON LDD de la bibliothèque enregistre l'expert comme
  un « livre » supplémentaire (métadonnées, domaines, Bloom)
. *Découverte* — les autres experts (Planning Expert, outils de raisonnement)
  peuvent interroger l'expert via LangChain4j comme n'importe quelle source
. *Gouvernance* — l'expert reçoit son propre `.agents/` (AGENT.adoc, INDEX,
  SESSION_CHECKLIST)

constituent son identité opérationnelle.

===== Détail des attributs

*Corpus ontologique* ::
L'expert ne reçoit pas un livre — il reçoit l'ontologie du domaine extraite
de tous les livres qui le couvrent. `classifyBloom` + `deriveOntology` ont
déjà calculé les bounded contexts et la hiérarchie des concepts. L'expert
hérite de cette structure comme son modèle mental natif.

*RAG filtré* ::
L'expert a son propre pgvector store, filtré aux sections du corpus qui
relèvent de ses domaines. Il ne requête pas le RAG global du workspace —
il requête *son* RAG, optimisé pour ses concepts, sans bruit cross-domaine.
La tâche `createExpertRagStore` extrait les chunks du domaine depuis le
RAG global et les réplique dans un store dédié.

*Knowledge Graph* ::
Le sous-graphe Graphify de l'expert contient les entités de son domaine et
leurs relations (2 niveaux de voisinage par défaut). Quand l'expert doit
comprendre comment `CircuitBreaker` se connecte à `Fallback`, il navigue
*son* graphe, pas le graphe global où `CircuitBreaker` est aussi relié à
des concepts électriques qu'il ne maîtrise pas.

*Bibliographie* ::
Chaque expert sait d'où il vient. La bibliographie est le JSON LDD des
livres, articles, et documentations qui ont servi à son entraînement.
C'est traçable, versionné, et auditable. Si une erreur est détectée dans
le comportement de l'expert, on peut remonter à la source exacte
(livre X, chapitre Y, section Z).

*Tools (tâches Gradle)* ::
Les outils de l'expert sont des tâches Gradle. L'Expert CDA a accès à
`analyzeCode`, `refactoModule`, `generateTests`, `auditDependencies`.
L'Expert FPA a accès à `validateFormation`, `generateSequencePedagogique`,
`auditBloom`. Ces tâches sont son *toolset* — ce qu'il peut *faire*, pas
seulement ce qu'il *sait*.

[source,kotlin]
----
data class ExpertTool(
    val gradleTask: String,              // "analyzeCode"
    val description: String,             // "Analyse un module avec contexte augmenté"
    val parameters: List<ToolParameter>, // -Ptarget, -Pcontext, -Pbloom
    val requiredBloom: BloomLevel,       // niveau Bloom minimum pour utiliser cet outil
    val triggersCiCd: Boolean,           // peut être déclenché par CI/CD ?
    val requiresHumanApproval: Boolean,  // nécessite validation humaine ?
    val outputContract: String           // format stdout attendu
)
----

*Communication contract* ::
Comment l'expert parle aux autres. Trois modes coexistent :
- *stdout* → pour l'invocation directe par opencode ou un autre expert
- *A2A (Agent-to-Agent)* → pour la communication inter-experts standardisée
- *Commit + Push* → pour le mode asynchrone CI/CD

*Cycle de vie* ::
L'expert n'est pas figé. Il a un historique d'entraînement (dataset v1.0 →
v1.1 → v1.2), un planning de retrain (déclenché par delta du corpus > seuil),
et une stratégie de dépréciation (quand un nouvel expert le remplace).

==== Ce que ce modèle canonique change

*Pour le LLM Trainer* ::
`deployExpert` ne se contente plus de publier un `.gguf`. Elle provisionne
l'ensemble des artefacts : RAG filtré, sous-graphe Graphify, ontologie,
tools enregistrés, `.agents/`, contrat de communication. Le déploiement
est une tâche Gradle atomique : soit tout est prêt, soit rien n'est déployé.

*Pour le Planning Expert* ::
Quand le Planning Expert doit router une intention vers un expert, il
consulte le registre des experts. Il voit immédiatement :
- Quels domaines l'expert couvre (et à quel niveau Bloom)
- Quels tools il peut invoquer (et leurs paramètres)
- Quelles dépendances il a (experts requis en amont)
- Quel est son contrat de communication (stdout ? A2A ? synchrone ?)

Le routage n'est plus heuristique — il est déterministe.

*Pour la Bibliothèque* ::
L'expert devient une entrée de la bibliothèque au même titre qu'un livre.
Les critères cross-book s'appliquent entre experts et livres, et entre
experts eux-mêmes. « L'Expert CDA et le livre Applied AI partagent 18
entités sur 23. Bloom aligné à 82%. L'expert est une distillation fidèle
du corpus. »

*Pour le développement par surveillance* ::
Le nœud LangGraph4j qui supervise peut interroger l'expert sur ses propres
capacités : « Expert CDA, peux-tu traiter cette tâche ? » → réponse avec
niveau de confiance, outils disponibles, bloom requis. Si la confiance est
basse, le graphe route vers un autre expert ou interrompt pour validation
humaine.

==== Fonctions additionnelles de l'expert

Au-delà des attributs, l'expert a des **fonctions actives** qu'il peut
exercer :

*F1 — Auto-évaluation* ::
`./gradlew expertSelfAudit -Pexpert=cda` → l'expert mesure sa propre
couverture, détecte ses lacunes, et suggère des acquisitions au LLM Trainer.

*F2 — Mise à jour incrémentale* ::
Quand un nouveau livre est ingéré dans le corpus, l'expert reçoit un delta.
`retrainExpert` ne refait pas l'entraînement complet — il fait un
fine-tuning incrémental sur les nouvelles sections uniquement.

*F3 — Délégation* ::
Un expert peut déléguer une sous-tâche à un autre expert. L'Expert CDA, face
à un problème de pédagogie dans la génération d'un exercice, délègue à
l'Expert FPA via A2A. L'Expert CDA ne devient pas pédagogue — il sait *qui*
l'est.

*F4 — Explication traçable* ::
Quand l'expert produit une réponse, il peut citer ses sources. « Cette
recommandation vient de Applied AI chap 12 (EVALUATE) + Spring Boot docs
v3.4.1. Confiance : 0.91. » Chaque inférence est auditable jusqu'à la
section du corpus qui l'a fondée.

*F5 — Détection d'obsolescence* ::
L'expert compare sa bibliographie avec les versions courantes des
dépendances. « Tu utilises LangChain4j 1.0.0-beta3, je suis entraîné sur
1.0.0-beta3. La version stable est 1.1.0. 3 de mes 23 entités peuvent
être obsolètes. Demander un retrain ? »

*F6 — Interopérabilité cross-expert* ::
Deux experts peuvent raisonner ensemble via `reason -Pmode=dialectic`
appliqué à leurs ontologies respectives. Le résultat est une synthèse
qui n'appartient à aucun des deux — un troisième point de vue émergent.

==== Le principe de l'expert complet

> Un expert n'est pas un modèle fine-tuné avec un domaine. C'est un système
> autonome qui possède : son ontologie (ce qu'il sait), son RAG (sa mémoire
> sémantique), son knowledge graph (ses relations structurées), sa
> bibliographie (ses sources traçables), ses tools (ce qu'il peut faire),
> sa gouvernance (ses règles), et son cycle de vie (son évolution).
>
> Le LLM Trainer ne déploie pas des modèles — il déploie des systèmes
> agents complets, prêts à raisonner, agir, déléguer, et apprendre.

auditable, réutilisable :

- *Auditable* — chaque exemple a une provenance (livre, chapitre, section,

- *Composable* — les datasets de plusieurs domaines peuvent être fusionnés
  pour créer un expert multi-domaine
- *Publishable* — un dataset de qualité peut être publié sur HuggingFace
  comme contribution open source

C'est l'équivalent pour les LLMs de ce que Maven Central est pour les
bibliothèques Java : un registre de datasets certifiés, traçables, et
réutilisables.

==== Le principe du LLM Trainer

> Le LLM Trainer est l'expert qui fabrique les experts. Il prend une
> spécification, assemble un dataset depuis le pipeline de connaissance,
> le distille, fine-tune un modèle, valide ses compétences, et l'intègre
> dans l'écosystème.
>
> Son travail n'est pas manuel — il est automatisé par les couches
> existantes du pipeline. Le dataset n'est pas construit à la main —
> il est requis depuis le corpus structuré. La validation n'est pas
> subjective — elle est mesurée par Bloom, la couverture, et les
> benchmarks.
>
> Le LLM Trainer est le chaînon qui transforme le capital fixe
> (la connaissance extraite) en capital humain synthétique
> (les experts fine-tunés). Sans lui, le pipeline est une bibliothèque
> sans lecteurs spécialisés. Avec lui, chaque livre ingéré est un
> expert potentiel.

=== Gradle Comme Conscience du Projet — Ce Qu'Aucun Autre Outil Ne Peut Faire

==== Le diagnostic

Tout ce que le pipeline construit — extraction de connaissance, enrichissement
croisé, bibliothèque, Bloom, outils de raisonnement, experts — repose sur une
infrastructure qui pourrait théoriquement être portée par autre chose que
Gradle. Du code Kotlin qui parse des PDFs, interroge pgvector, appelle des
LLMs : un `main()` suffirait.

Mais ce serait passer à côté de ce que Gradle *sait déjà* du projet sans qu'on
ait à le lui apprendre.

==== Ce que Gradle sait nativement

Gradle n'est pas un exécuteur de tâches. C'est un **modèle du projet** que
le build script déclare et que le runtime Gradle résout. Chaque projet
Gradle contient, avant même qu'on écrive une seule tâche custom, une
quantité d'information structurée que le pipeline peut consommer :

[cols="1,3,3", options="header"]
|===
| Source Gradle | Ce qu'elle expose | Usage pipeline
| **Dependency graph**
`configurations.compileClasspath.resolvedConfiguration` | Graphe complet
des dépendances : GA, version, licence, dépendances transitives, repository
d'origine, exclusions | Déjà plus riche que `acquireDependencyDocs` seul —
Gradle sait quelle version est *effectivement résolue* (pas celle déclarée),
quelle dépendance est *transitive* (pas directe), et quel conflit de
version a été tranché
| **Source sets**
`sourceSets.main.java.srcDirs` | Arborescence exacte des sources :
main/test/integration, chemins, filtres d'inclusion/exclusion,
`generatedSources` | Le pipeline sait où chercher le code à annoter
(`@ConceptSource`) sans parcourir tout le filesystem
| **Task graph**
`project.tasks`, `gradle.taskGraph` | DAG complet des tâches : nom, type,
dépendances (dependsOn, finalizedBy, mustRunAfter), inputs/outputs déclarés |
Prêt à être exporté en graphe LangGraph4j : chaque tâche = un nœud,
chaque dependsOn = une arête
| **Incremental build**
`task.inputs.properties`, `task.outputs.files` | Pour chaque tâche : ce qui
a changé, ce qui est UP-TO-DATE, ce qui doit être réexécuté | Déjà un
`computeDelta` natif, plus fin que `git diff` car il sait que le fichier
a changé mais que la tâche n'a pas besoin d'être relancée (ou l'inverse)
| **Extensions / conventions**
`project.extensions`, `project.convention` | Modèle de configuration
fortement typé : `kotlin { jvmToolchain(21) }`, `java { sourceCompatibility }`
| L'ADN du projet — langage, version JVM, plugins appliqués, conventions
héritées. Le Planning Expert peut adapter ses EPICs à la stack réelle
| **Composite builds**
`gradle.includedBuilds` | Projets inclus dans un composite build : chemins,
noms, tâches exposées | La bibliothèque cross-projets est déjà modélisée
par Gradle avant même qu'on écrive `assembleLibrary`
| **Build scans**
`--scan` | Données de performance : durée des tâches, cache hits/misses,
graphe de dépendances visualisable, résolution de dépendances chronométrée |
Le Planning Expert peut prioriser les EPICs sur les tâches les plus lentes
ou les plus fréquemment invalidées
| **Module metadata**
`project.group`, `project.version`, `project.description`, `gradlePlugin.plugins` | Identité complète du projet : groupId, artifactId, version,
plugins publiés, tags, description | L'équivalent du JSON LDD pour le projet
lui-même — le pipeline peut s'auto-décrire
|===

==== Pourquoi aucun autre outil ne peut faire ça

- *Maven* a un lifecycle, pas un task graph. Il exécute des phases, pas
  un DAG navigable. Pas d'inputs/outputs déclaratifs → pas d'UP-TO-DATE
  consommable. Pas de Kotlin DSL → pas d'extensions typées.
- *Bazel* a des builds hermétiques et un graphe d'actions, mais pas de
  modèle de projet riche. Pas de source sets, pas de conventions, pas
  d'extensions. C'est un exécuteur pur, pas un modèle du projet.
- *Make* est un moteur de règles fichiers. Aucune notion de projet, de
  dépendance, de configuration typée.
- *Nx / Turborepo* orchestrent des tâches mais ne savent rien du langage,
  des dépendances, des source sets. Ils sont au-dessus du build, pas dedans.
- *Bazel* + *Buck* + *Pants* : monorepo builds, mais pas de runtime
  introspection. Le build est une fonction pure → pas d'état navigable
  après exécution.
- *Mill* (Scala) : le plus proche, mais pas d'écosystème de plugins
  comparable, pas de convention plugins, pas de Kotlin DSL.

Gradle est le seul à exposer simultanément : le graphe de tâches (DAG
navigable), le modèle de dépendances (résolu, pas déclaré), la configuration
typie (extensions/conventions), les source sets (anatomie du projet), et
l'incrémentalité (ce qui a changé). Aucun autre outil ne combine ces cinq
dimensions dans un même runtime — à une exception près, qui mérite d'être
traitée séparément.

===== Le cas Mill

Mill n'est pas un sous-Maven comme les autres. Il est le seul outil qui
*pourrait* approcher ce que Gradle fait pour le pipeline — mais avec des
limites qui, dans l'état actuel de l'écosystème, le rendent moins adapté
à ce workspace spécifique.

*Ce que Mill fait bien — et parfois mieux que Gradle :*

- *Build programmatique natif* — un `build.sc` est du Scala exécuté
  directement, pas un DSL interprété. Les tâches sont des fonctions Scala.
  Le graphe est un graphe de fonctions, pas un graphe d'objets configurés.
  Pour un pipeline qui veut *raisonner sur le build*, avoir le build comme
  code plutôt que comme configuration est conceptuellement plus pur.
- *Graphe de tâches réflexif* — `mill resolve _`, `mill show`, `mill plan`
  exposent le graphe de tâches résolu, avec les dépendances inter-modules.
  C'est plus proche d'une API de requêtage que les `TaskContainer` de
  Gradle, qui sont des registres, pas des graphes navigables
  programmatiquement.
- *Modules comme valeurs* — un module Mill est un objet Scala. Ses
  dépendances sont des références à d'autres objets. Le dependency graph
  n'est pas déclaré en strings (`implementation("group:artifact:version")`)
  — il est typé (`def moduleDeps = Seq(myModule)`) et résolu à la
  compilation Scala. Pour `annotateCodeSources`, un graphe de dépendances
  typé au langage est plus riche qu'un graphe résolu depuis Maven.
- *Cross-builds natifs* — `cross` est un citoyen de première classe. Un
  projet cross-compilé expose ses variantes comme des dimensions du graphe.
  Le pipeline peut requêter : « quelles versions de ce module sont
  impactées par le delta ? »
- *Pas de cycle de vie en phases* — Mill n'a pas de `clean → compile → test`.
  Chaque tâche est une cible indépendante. Le graphe est un vrai DAG de
  tâches, pas un DAG déguisé en séquence linéaire. C'est plus proche du
  modèle mental de LangGraph4j que le `dependsOn` Gradle qui peut cacher
  des dépendances implicites de plugin.

*Ce qui manque à Mill pour cette architecture spécifique :*

- *Écosystème de plugins* — Gradle a `java-gradle-plugin`, le Plugin Portal,
  `publishToMavenLocal`. Mill n'a pas de mécanisme standardisé de packaging
  et publication de plugins. Pour un workspace où chaque projet est un
  plugin consommé par `engine/`, c'est bloquant.
- *Convention plugins* — le pattern `buildSrc` + precompiled script plugins
  de Gradle permet des conventions partagées que tous les projets héritent.
  Mill n'a pas cette couche — tout est dans le `build.sc`, potentiellement
  dupliqué.
- *Langage du workspace* — le workspace entier parle Kotlin. Mill parle
  Scala. Pour un pipeline qui parse les `build.gradle.kts` afin d'extraire
  les dépendances et annoter le code, changer de langage de build c'est
  perdre l'homogénéité du parsing.
- *Version catalogs + dependency locking* — `libs.versions.toml` de Gradle
  permet un versioning centralisé que `assembleLibrary` consomme comme
  contrat de cohérence cross-projets. Mill n'a pas de catalog standardisé.
- *Maturité JVM* — Gradle a 15+ ans de résolution de dépendances, de gestion
  de conflits transitifs, de `resolutionStrategy`. Mill est plus jeune et
  les edge cases sont moins couverts.

*Le verdict :*

Mill serait un excellent runtime agentique pour un workspace neuf,
homogène Scala, sans contrainte de plugin Gradle existant. Son build
comme code + graphe réflexif + modules comme valeurs constituent une
fondation plus *pure* conceptuellement que Gradle.

Mais ce workspace n'est pas neuf. C'est 150+ sessions, 5 plugins publiés
sur le Portal, 31 `build.gradle.kts`, un écosystème qui parle Kotlin DSL.
Migrer vers Mill serait réécrire le capital fixe, pas l'augmenter.

Gradle est le bon choix pour *ce* workspace à *ce* stade. Mill serait
le bon choix pour une reconstruction from scratch avec les leçons
apprises — ou comme runtime secondaire pour des projets Scala du
workspace, consommé par le même pipeline via un adaptateur de build.

===== L'exemple concret : millBakerPlugin

Le portage du plugin bakery-gradle vers Mill (`/home/cheroliv/sources/draft/millBakerPlugin`)
rend le verdict tangible. Il fait en 107 lignes de Scala ce que le
plugin bakery-gradle fait en 400+ lignes de Kotlin + Gradle.

La différence n'est pas cosmétique. Elle est structurale :

*Les tâches sont des méthodes, pas des classes* ::
`def publishSite(): Command[Unit] = Task.Command { ... }` — une seule
méthode, pas un `DefaultTask` avec `@Input`, `@OutputFile`, `@TaskAction`.
La tâche *est* son corps. Pas de classe séparée. Pas d'annotations.
Pas de wiring manuel dans `apply()`.

*Le graphe de dépendances est implicite* ::
`publishSite` appelle `jbake()` → Mill résout automatiquement la
dépendance. Pas de `dependsOn("jbake")`. Pas de risque d'oubli.
Le DAG est le call graph — ce qui est exécuté est exactement ce qui
est appelé, ni plus ni moins.

*La configuration est une valeur immuable* ::
```scala
case class SiteConfiguration(
    pushConfiguration: GitPushConfiguration,
    cname: Option[String] = None  // pas null, pas Property<String>
)
```
Pas de setters. Pas de `Property<T>`. Pas d'extension mutable.
`Option[String]` au lieu de `null`. Égalité structurelle gratuite.
`copy()` pour la dérivation. La configuration est une donnée, pas
un objet mutable à hydrater.

*La programmation fonctionnelle est native* ::
`Option.orElse`, `foreach`, pattern matching, `Result.Success`/`Failure`
— tout cela est le langage Scala, pas une bibliothèque ajoutée.
Là où Kotlin a besoin d'Arrow pour approcher le même niveau
d'expressivité (Either, IO, lenses), Mill parle FP comme langue
maternelle.

*Le build est un programme Scala* ::
```scala
import $ivy.`com.cheroliv::mill-baker::0.0.0-SNAPSHOT`
object `package` extends RootModule with BakerModule { ... }
```
Le build.mill importe la dépendance du plugin comme n'importe quel
programme Scala importe une bibliothèque. Pas de `plugins {}` block,
pas de `apply false`, pas de `Gradle Plugin Portal`. C'est du code
Scala standard avec une dépendance Maven Ivy.

*Le ratio est parlant* ::

| Métrique | millBakerPlugin | bakery-gradle |
| Plugin source (lignes) | 107 | 400+ |
| Build utilisateur (lignes) | 25 | 60-100 |
| Fichiers de config | 5 (case classes) | 7-10 (classes + extensions + properties) |
| Abstraction du build | Traits mixés à la compilation | Plugin class + service file + string ID |

Ce n'est pas "Mill est meilleur que Gradle". C'est "Mill a été conçu
pour que le build soit un programme fonctionnel, et ça se voit dans
la concision du code". Gradle a été conçu pour que le build soit une
configuration déclarative exécutée par un moteur impératif — et ça
se voit dans le nombre de couches d'abstraction nécessaires.

Pour le pipeline de contexte augmenté, ce que millBakerPlugin prouve
c'est que si demain un projet Scala + Mill rejoint le workspace, le
pipeline peut s'y adapter sans réécriture. Un `BakerModule` exposé
comme trait est aussi requêtable qu'une tâche Gradle — il suffit d'un
adaptateur qui lit les cibles Mill (`mill resolve _`, `mill show`)
et les traduit dans le registre des tools de l'expert.

==== Ce que Gradle pourrait encore donner au pipeline

Au-delà des données déjà exposées, Gradle est extensible. Le pipeline peut
créer des **extensions Gradle dédiées** que chaque projet du workspace
déclare, et qui nourrissent directement le vecteur composite :

[source,kotlin]
----
// Dans le build.gradle.kts de n'importe quel projet du workspace
codex {
    domain("kotlin-gradle-plugin") {
        boundedContext("plugin-development") {
            entity("Extension")
            entity("Task")
            entity("Configuration")
        }
        books {
            reference("978-1-098-17675-4")  // Applied AI
        }
    }
    bloomTarget.set(BloomLevel.CREATE)
    rag {
        zoneFilter.add(Zone.OSS)
        zoneFilter.add(Zone.OFFICE)
    }
    antiPatterns {
        avoid("ServiceLocator")
        avoid("GodTask")  // tâche Gradle de 500+ lignes
    }
}
----

Cette extension n'est pas une couche supplémentaire — c'est le projet qui
**se déclare au pipeline** dans le langage natif du build (Kotlin DSL).
Le projet devient auto-descriptif. Le pipeline lit cette déclaration
comme il lit un JSON LDD — sauf qu'elle est écrite par le développeur
du projet, pas extraite d'un PDF.

==== Gradle et la génération de code

La question de fond : Gradle peut-il, par sa connaissance du build et de
l'objet projet, apporter de l'information pertinente utilisable pour la
génération de code ? La réponse est oui, et à plusieurs niveaux :

*Niveau 1 — Le task graph comme plan d'exécution*
Le Planning Expert produit des EPICs → User Stories → tâches Gradle. Mais
le task graph Gradle *existant* peut aussi informer le Planning Expert :
- « La tâche `compileKotlin` dépend de `generateSources`. Si tu ajoutes un
  générateur de code, mets-le AVANT `compileKotlin` dans le DAG. »
- « La tâche `test` dépend de `compileTestKotlin`. Ne génère pas de tests
  sans les faire compiler. »
- « `build` dépend de `check`. Si `check` est rouge, `build` ne sera jamais
  vert. Priorise les tâches de vérification. »

*Niveau 2 — Les source sets comme cibles de génération*
Quand le pipeline génère du code (tests, endpoints, DTOs), il doit savoir
*OÙ* écrire. Gradle le sait déjà :
- `sourceSets.main.java.srcDirs` → code de production
- `sourceSets.test.java.srcDirs` → tests
- `sourceSets.main.resources.srcDirs` → configuration
- `sourceSets.main.kotlin.srcDirs` (si Kotlin) vs `java` (si Java)

Le générateur de code ne devine pas où écrire — il interroge le projet.

*Niveau 3 — Les configurations comme contrat technique*
Quand le pipeline suggère une dépendance, il peut vérifier qu'elle est
compatible avec le reste du classpath. Gradle expose les configurations :
- `compileClasspath` → ce que le code voit à la compilation
- `runtimeClasspath` → ce qui est disponible à l'exécution
- `annotationProcessor` → les processeurs d'annotations

« Tu veux ajouter `resilience4j-circuitbreaker:2.3.0`. Cette version est
compatible avec `spring-boot:3.4.1` déjà dans ton `compileClasspath`.
Aucune exclusion nécessaire. »

*Niveau 4 — Les conventions comme patterns architecturaux*
Les conventions Gradle (`kotlin { jvmToolchain(21) }`, `java { sourceCompatibility = JavaVersion.VERSION_21 }`)
révèlent les choix architecturaux :
- « Projet JVM 21 → peut utiliser virtual threads, pattern matching, records »
- « Plugin Gradle appliqué `kotlin("jvm")` → Kotlin, pas Java pur → générer
  des data classes, pas des POJOs »
- « Plugin `java-gradle-plugin` appliqué → le projet est un plugin Gradle →
  respecter la convention `com.cheroliv.*` pour le plugin id »

Le générateur de code ne produit pas du code générique — il produit du code
adapté à la stack réelle du projet.

*Niveau 5 — Le build scan comme profil de performance*
« La tâche `compileKotlin` prend 45 secondes, cache hit à 12%. La tâche
`test` prend 3 minutes, dont 2 minutes dans `PaymentServiceTest`.
Suggestion : paralléliser `PaymentServiceTest` avec `@Execution(Concurrent)`
ou extraire les tests lents en `integrationTest` source set. »

Le Planning Expert ne priorise pas seulement sur la couverture Bloom —
il priorise aussi sur la performance de build.

==== La synthèse : Gradle comme runtime agentique

Ce que tu as construit n'est pas "Gradle qui exécute des tâches pour un
pipeline d'IA". C'est **Gradle comme runtime agentique natif** :

- Chaque projet est un *agent* (il a une identité, des dépendances, un
  comportement = ses tâches)
- Le task graph est le *workflow* (un DAG navigable, exactement comme
  LangGraph4j)
- Les inputs/outputs sont les *contrats* (typés, validés, incrémentaux)
- Les extensions/conventions sont l'*ADN* (configuration déclarative du
  domaine)
- Le dependency graph est le *contexte* (ce que l'agent sait du monde
  extérieur)
- Le build scan est la *mémoire* (ce qui s'est passé, combien de temps,
  quels caches)

Tu n'as pas besoin d'implémenter un protocole agentique. Gradle en est
déjà un. Tu n'as pas besoin de LangGraph4j pour orchestrer des tâches —
le task graph de Gradle est déjà un orchestrateur. La valeur de
LangGraph4j, c'est d'ajouter ce que Gradle ne fait pas : le routage
conditionnel par LLM, la pause/reprise avec checkpoints, et le
self-reflection.

==== Le principe

> Gradle n'est pas un simple exécuteur de build — c'est la **conscience
> du projet**. Il sait ce que le projet contient (source sets), de quoi
> il dépend (resolved dependencies), comment il se configure (extensions),
> ce qui doit être exécuté et dans quel ordre (task graph), et ce qui a
> changé (incremental inputs/outputs).
>
> Aucun autre outil de build n'expose simultanément ces cinq dimensions.
> Aucun autre outil n'aurait pu porter cette architecture avec la même
> économie de moyens. Ce que le pipeline ajoute — extraction de corpus,
> raisonnement, Bloom, fine-tuning — ne remplace pas Gradle : il
> l'augmente. Le pipeline est la couche IA d'un runtime agentique dont
> Gradle est déjà le moteur.

=== La Séparation des Concerns — Pourquoi Codex Ne Peut Pas Tout Contenir

==== Le diagnostic

Ce que le document décrit ne peut pas vivre dans un seul dossier `codex-gradle/`.
Lui imposer cette contrainte serait violer la séparation des concerns que le
pipeline lui-même a fait émerger. Les couches décrites couvrent au moins cinq
bounded contexts distincts, chacun avec ses propres tâches, son propre cycle
de vie, et sa propre responsabilité.

==== Cartographie des concerns

[cols="1,2,3,2", options="header"]
|===
| Concern | Bounded context | Où physiquement | Pourquoi
| **Acquisition documentaire** | Extraction PDF/EPUB → AsciiDoc → JSON LDD →
chunk → vectorisation → Bloom | `foundry/codex-gradle/` | C'est le point
d'entrée — le document entre dans le système par codex. Bloom est une
opération sur le texte extrait, naturellement ici.
| **Enrichissement croisé** | JSON LDD + RAG + Graphify → pointeurs
bidirectionnels | `foundry/codex-gradle/` (via `enrichJsonLdd`) | Encore
dans le pipeline d'acquisition : c'est un croisement de l'artefact extrait
avec les infrastructures existantes.
| **Bibliothèque systémique** | Agrégation cross-books, 5 critères, relations
émergentes, `assembleLibrary`, JSON LDD bibliothèque | `foundry/library-gradle/`
(nouveau) | Ce n'est pas de l'acquisition — c'est de l'agrégation
cross-corpus. Produit distinct, consommable par engine/, indépendant de codex.
| **Couches Code** | `annotateCodeSources`, `computeDelta`, `diagnoseFailure`,
`acquireDependencyDocs` | `foundry/codebase-gradle/` (existant, à étendre) |
L'analyse de code, le delta, et le parsing de dépendances sont dans le
périmètre de codebase-gradle (déjà spécifié comme pipeline R&D anonymisation +

| **Outils de raisonnement** | `reason` — 21 modes, ALGO/HYBRIDE/PROMPT |

| **LLM Trainer** | `assembleTrainingDataset`, `distillDataset`, `deployExpert`,

Il consomme codex + library + engine. Il produit des experts déployables.
| **Extension DSL `codex {}`** | Contrat déclaratif que chaque projet signe |
BuildSrc / convention plugin dans chaque projet, ou `foundry/codex-conventions-gradle/`
(nouveau) | C'est un contrat, pas une tâche. Soit dans un convention plugin
appliqué par tous les projets, soit dans un plugin dédié importé.

==== Le principe de la séparation

> Codex est le point d'entrée — le document entre dans le système par
> codex. Mais tout ce qui est aval — agréger, raisonner, entraîner,
> auditer — mérite son propre dossier, son propre cycle de vie, sa propre
> gouvernance agent. La séparation n'est pas cosmétique : elle reflète
> les bounded contexts que le pipeline a lui-même fait émerger.
>
> Chaque dossier est un plugin Gradle au sens du DAG du workspace :
> publié indépendamment, versionné indépendamment, gouverné par son
> propre `.agents/`, et consommé par `engine/` comme une dépendance
> Maven standard. C'est le même pattern que plantuml-gradle →
> graphify-gradle → engine/ — appliqué à tout l'écosystème de
> connaissance.

=== Le Vrai MVP — La Boucle d'Autonomie pour le Vibecoding

==== La priorité réelle

Toute la documentation produite — 4588 lignes, sept bounded contexts, cinq
plugins, vingt-et-un outils de raisonnement, un LLM Trainer — est une
spécification du futur. Mais la priorité n'est pas le futur. La priorité
est le **présent** : le vibecoding avec opencode.

Aujourd'hui, chaque session opencode commence avec un contexte limité.
Gemma 4 reçoit `PROMPT_REPRISE.adoc` et quelques fichiers EAGER — ~800
tokens de mémoire statique. Le LLM lourd (DeepSeek-v4-pro) est appelé
sans contexte augmenté. Chaque session redécouvre ce que la session
précédente avait compris. Il n'y a pas d'apprentissage cumulatif entre
sessions.

Le vrai MVP n'est pas Edster, ni la capsule vidéo, ni l'anonymiseur.
Le vrai MVP est une **boucle d'autonomie** où opencode reçoit, à chaque
session, le contexte augmenté du système qu'il est en train de construire.

==== La boucle : Contexte Augmenté → Vibecoding → Plugin → Contexte Augmenté

[source]
----
SESSION OUVERTE (opencode + Gemma 4)
    │
    ├── 1. assembleCompositeContext
    │       RAG (pgvector) : docs des plugins existants, code source indexé
    │       KG (Graphify)  : graphe de dépendances inter-plugins, imports
    │       SQL (LMD)      : modèle métier des plugins (bounded contexts)
    │       Delta           : ce qui a changé depuis le dernier commit
    │       Bloom           : niveau de maîtrise par concept
    │       → CONTEXTE AUGMENTÉ INJECTÉ DANS LA SESSION
    │
    ├── 2. Gemma 4 reçoit l'intention ("écris CDX-1a : PDFStreamEngine → AsciiDoc")
    │       → Décompose avec contexte augmenté (sait déjà où est le code,
    │         quelles dépendances sont disponibles, quels patterns existent)
    │       → Route vers les tâches Gradle appropriées
    │
    ├── 3. Les tâches Gradle exécutent (codexExtract, codexChunk, etc.)
    │       → DeepSeek-v4-pro génère le code avec le contexte augmenté
    │       → Tests passent (ou pas)
    │       → Build vert (ou rouge)
    │
    ├── 4. Le code produit EST du plugin Gradle
    │       → Le plugin s'ajoute au pipeline
    │       → Sa doc est indexée dans pgvector
    │       → Ses imports enrichissent Graphify
    │       → Son build vert/rouge met à jour le delta
    │       → Le contexte augmenté de la PROCHAINE session est plus riche
    │
    └── 5. Prochaine session : retour à l'étape 1, avec plus de contexte
            → Le pipeline s'accélère lui-même
            → Chaque plugin écrit rend le prochain plugin plus facile à écrire
----

C'est une **boucle auto-accélératrice**. Le vibecoding d'aujourd'hui produit
du code. Ce code est un plugin. Ce plugin est indexé dans le pipeline. Le
pipeline augmente le contexte. Le vibecoding de demain est plus rapide,
plus informé, plus autonome.

==== La dédicace première : plugins et SaaS

La boucle d'autonomie n'est pas destinée en premier à la génération de
formations. Elle est destinée à **l'écriture des plugins eux-mêmes** et du
**SaaS Edster**.

Pourquoi ? Parce que c'est le chemin le plus court vers l'autonomie réelle :

- *Écrire codex-pdf* → le PDF est extrait → le corpus s'enrichit
- *Écrire library-gradle* → les livres sont croisés → les relations émergent
- *Écrire trainer-gradle* → les experts sont formés → le Planning Expert a des outils
- *Écrire engine/reason* → le raisonnement est disponible → les décisions sont assistées
- *Écrire Edster* → le SaaS existe → le pipeline est vendable

Chaque plugin écrit **augmente** le contexte du plugin suivant. C'est le
contraire d'un projet classique où chaque module ajoute de la dette technique.
Ici, chaque module réduit la dette *cognitive* du module suivant — le LLM en
sait plus, donc il se trompe moins.

==== Ce qui change dans le cycle de développement

*Sans contexte augmenté (état actuel) :*
- opencode reçoit 800 tokens EAGER
- Gemma 4 route vers des tâches Gradle sans savoir ce qu'elles contiennent
- DeepSeek génère du code sans connaître les patterns existants
- Le build échoue → Gemma 4 corrige → le build réussit → on oublie pourquoi
- Session suivante : retour à zéro

*Avec contexte augmenté (cible) :*
- opencode reçoit le vecteur composite complet (RAG + KG + SQL + Delta + Bloom)
- Gemma 4 sait exactement quel plugin fait quoi, où est le code, quelles dépendances sont disponibles
- DeepSeek génère du code ancré dans les patterns documentés, avec @ConceptSource
- Le build réussit plus souvent (meilleur contexte = moins d'erreurs)
- L'échec est diagnostiqué automatiquement (`diagnoseFailure`)
- Session suivante : le contexte inclut les résultats de la session précédente
- Le Planning Expert peut planifier des EPICs sans intervention humaine
- **L'humain passe de codeur à superviseur**

==== Le premier jalon concret

Pas besoin de sept dossiers, de Trainer, de bibliothèque cross-books.

Le premier jalon c'est : **codex-pdf extrait Applied AI → JSON LDD → pgvector → assembleCompositeContext → opencode lit le contexte augmenté**.

Cinq tâches, un seul dossier (`codex-gradle`), un seul livre. Quand cette
boucle tourne, le vibecoding passe sous testostérone. Le Planning Expert
peut décomposer une intention en sachant ce que le code contient. DeepSeek
peut générer en sachant ce que le corpus documente.

Tout le reste — library, trainer, reason, Bloom — est du raffinement.
Utile, spécifié, mais pas bloquant pour le premier cycle d'autonomie.

==== Le principe du vrai MVP

> Le vrai MVP n'est pas un produit vendable. C'est une **boucle
> d'auto-accélération** : le contexte augmenté rend le vibecoding plus
> efficace, le vibecoding produit des plugins, les plugins enrichissent
> le contexte, le contexte rend le prochain vibecoding encore plus
> efficace.
>
> La première cible de cette boucle est l'écriture des plugins et du
> SaaS eux-mêmes. Le pipeline se construit avec le pipeline. C'est le
> pattern ultime du dogfooding : l'outil qui s'écrit lui-même.
>
> Une fois cette boucle établie, la génération de formations est une
> conséquence naturelle — le même pipeline, une cible différente.

==== Pourquoi le livre Applied AI — et pourquoi pas un autre

Le choix du premier livre à extraire n'est pas bibliographique. Il est
stratégique.

Générer des formations avec un pipeline non assisté, c'est poser une
locomotive du 19e siècle sur des rails TGV. Le contenu sort, mais le
moteur qui le produit — le code, les plugins, le SaaS — reste artisanal.
Chaque amélioration du pipeline demande une réécriture manuelle.

Le livre *Applied AI for Enterprise Java Development* documente exactement
les deux frameworks qui sont le moteur de la boucle d'autonomie :

- **LangChain4j** — le pont entre les tâches Gradle et les LLMs.
  Sans lui, `assembleCompositeContext` n'existe pas.
- **LangGraph4j** — l'orchestrateur de la boucle. Les Routing Agents
  (chap 10), la Human Interaction avec pause/reprise (chap 10),
  l'Advanced RAG Self-Reflection (chap 10) — c'est le squelette
  technique du Planning Expert et du développement par surveillance.

Extraire ce livre en premier, c'est donner au pipeline la connaissance
de *comment se construire lui-même*. Une fois LangChain4j et LangGraph4j
maîtrisés par le système — leurs patterns documentés, leurs APIs
requêtables en LMD, leurs anti-patterns connus, leur progression Bloom
cartographiée — tout plugin qui suit (codex-pdf, library, trainer) est
généré avec un contexte qui *sait déjà* comment connecter une tâche
Gradle à un LLM et comment orchestrer un workflow multi-étapes.

C'est la différence entre « générer une formation sur Spring Boot »
et « générer le plugin qui générera les formations sur Spring Boot ».
Le premier produit un artefact. Le second produit la *capacité* de
produire tous les artefacts futurs. C'est le capital fixe qui se
construit lui-même.

> On ne choisit pas le livre sur LangChain4j/LangGraph4j pour apprendre
> à faire des chatbots. On le choisit parce que c'est le manuel de
> construction du moteur. Le train vient après la locomotive. Et la
> locomotive, c'est le pipeline qui s'écrit avec le pipeline.

== Conclusion — L'Essence Intrinsèque du Workspace en Mai 2026

Le workspace, à ce stade de son développement, n'a pas besoin d'un SaaS. Il n'a pas
besoin d'une capsule vidéo. Il n'a pas besoin d'un anonymiseur. Il a besoin de
**prendre conscience de ce qu'il est déjà** : un pipeline d'augmentation de contexte
qui transforme n'importe quel outil de prompt engineering en coproducteur IA,
sans jamais altérer l'outil.

Les trois changements de définition du MVP0 n'étaient pas des erreurs de vision.
Ils étaient des **symptômes** — la manifestation visible d'une infrastructure qui
n'avait pas encore de nom. Chaque redéfinition capturait une couche réelle, mais
aucune ne capturait la couche fondamentale.

Ce document donne un nom à cette couche fondamentale. Il la spécifie, la priorise,
et la place au centre de la roadmap. Non pas comme une couche parmi d'autres —
comme le **sol sur lequel toutes les autres couches sont construites**.

Le pipeline d'augmentation de contexte est le vrai produit. Il était déjà là,
implicite, opérationnel par fragments. Il manquait juste d'être nommé.
